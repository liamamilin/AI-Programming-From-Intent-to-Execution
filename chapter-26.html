27  Performance & Cost

27.1 From Correctness to Efficiency

So far, we have focused on:

  • Defining systems

  • Achieving correctness

  • Ensuring reliability

However, a system that is correct but inefficient is not practical.

Real systems must also consider:

Performance and Cost

27.2 The Three Core Metrics

In LLM systems, performance is primarily determined by three factors:

  • Token Usage

  • Latency

  • Cost

These are tightly coupled.

27.2.1 Key Relationship

More Tokens → Higher Latency → Higher Cost

Efficiency is about controlling these three dimensions.

27.3 What Are Tokens?

Tokens are the basic unit of processing for language models.

They represent:

  • Words

  • Parts of words

  • Symbols

27.3.1 Example

Sentence:

“This is a test”

May be split into:

  • “This”

  • “is”

  • “a”

  • “test”

27.3.2 Why Tokens Matter

  • Models process tokens, not words

  • Pricing is based on token usage

  • Latency increases with token count

Tokens are the currency of LLM systems.

27.4 Token Usage and System Design

Token usage comes from:

  • Input context

  • Instructions

  • Output generation

27.4.1 Sources of Token Growth

27.4.1.1 1. Long Context

  • Large prompts

  • Extensive memory

27.4.1.2 2. Verbose Instructions

  • Unnecessary detail

  • Redundant text

27.4.1.3 3. Large Outputs

  • Long responses

  • Unstructured generation

Poor design leads to token waste.

27.5 Latency

Latency is the time it takes for the system to:

  • Receive input

  • Produce output

27.5.1 Components of Latency

  • Model processing time

  • Network delay

  • Tool execution time

27.5.2 Why Latency Matters

  • User experience

  • System responsiveness

  • Real-time applications

Fast systems feel intelligent.
Slow systems feel broken.

27.6 Cost

Cost is directly tied to:

  • Token usage

  • Number of calls

  • Model type

27.6.1 Cost Drivers

  • Large context windows

  • Frequent API calls

  • Complex pipelines

27.6.2 Implication

Even small inefficiencies:

  • Scale significantly

  • Increase operational cost

Cost is a function of design, not just usage.

27.7 Trade-offs Between Metrics

27.7.1 Token vs Accuracy

  • More tokens → more context → better accuracy

  • But higher cost

27.7.2 Latency vs Quality

  • Faster responses → less processing

  • But potentially lower quality

27.7.3 Cost vs Reliability

  • More checks → higher cost

  • But better stability

System design requires balancing trade-offs.

27.8 Optimization Strategies

27.8.1 1. Reduce Token Usage

  • Compress context

  • Remove redundancy

  • Use summaries

27.8.2 2. Optimize Instructions

  • Keep concise

  • Avoid unnecessary text

27.8.3 3. Control Output Length

  • Define limits

  • Use structured output

27.8.4 4. Efficient Context Management

  • Retrieve only relevant data

  • Avoid full history

27.8.5 5. Cache Results

  • Reuse previous outputs

  • Avoid repeated computation

Efficiency comes from better design, not shortcuts.

27.9 Architectural Optimization

27.9.1 1. Pipeline Design

  • Break tasks into smaller steps

  • Avoid overloading single calls

27.9.2 2. Selective Execution

  • Only run necessary components

27.9.3 3. Model Selection

  • Use smaller models where possible

  • Reserve larger models for complex tasks

Not all tasks require maximum capability.

27.10 Measuring Performance

To optimize, you must measure.

27.10.1 Key Metrics

  • Tokens per request

  • Latency per step

  • Cost per task

27.10.2 Approach

  • Log system behavior

  • Analyze bottlenecks

  • Iterate design

Measurement enables optimization.

27.11 Common Mistakes

27.11.1 1. Ignoring Token Usage

  • Leads to unexpected costs

27.11.2 2. Overloading Context

  • Reduces efficiency

  • Increases latency

27.11.3 3. Over-Engineering Pipelines

  • Too many steps

  • Increased cost

27.11.4 4. Optimizing Too Early

  • Premature optimization

  • Reduced flexibility

27.12 Performance as a System Property

Performance is not determined by a single component.

It depends on:

  • Spec design

  • Instruction efficiency

  • Context management

  • System architecture

Efficiency emerges from the entire system.

27.13 Final Insight

In the LLM era, performance is not just about speed.

It is about:

  • Efficient use of tokens

  • Balanced latency

  • Controlled cost

The best systems are not just correct.
They are efficient.

27.14 Closing Statement

As systems scale, performance and cost become critical constraints.

A well-designed system delivers value
not just correctly, but efficiently.

27  Performance & Cost

27.1 From Correctness to Efficiency

So far, we have focused on:

  • Defining systems—creating clear specifications
  • Achieving correctness—ensuring systems produce right outputs
  • Ensuring reliability—making systems consistently accurate

However, a system that is correct but inefficient is not practical. A system that produces perfect results but is too slow or too expensive cannot be used in production.

Real systems must also consider:

Performance and Cost

27.2 The Three Core Metrics

In LLM systems, performance is primarily determined by three interrelated factors:

  • Token Usage—how much text the system processes and generates
  • Latency—how long the system takes to respond
  • Cost—how much money or resources the system consumes

These metrics are tightly coupled—they influence each other and must be balanced together.

27.2.1 Key Relationship

More Tokens → Higher Latency → Higher Cost

Processing more text takes more time and costs more money. Optimizing one metric often affects the others.

Efficiency is about controlling these three dimensions simultaneously. You cannot optimize one in isolation.

27.3 What Are Tokens?

Tokens are the basic unit of processing for language models. They represent the chunks of text that models read and generate.

They can be:

  • Words—complete words in many cases
  • Parts of words—subword units for complex or rare words
  • Symbols—punctuation, numbers, special characters

27.3.1 Example

Sentence: “This is a test”

May be tokenized as: - “This” - “is” - “a” - “test”

Or possibly as: - “This” - ” is” - ” a” - ” test”

The exact tokenization depends on the model and its tokenizer.

27.3.2 Why Tokens Matter

  • Models process tokens, not words—computational cost is per-token
  • Pricing is based on token usage—most LLM APIs charge per token
  • Latency increases with token count—more tokens mean more processing time

Tokens are the currency of LLM systems. Every operation has a token cost.

27.4 Token Usage and System Design

Token usage comes from multiple sources in the system:

  • Input context—information provided to the system
  • Instructions—the protocol defining execution
  • Output generation—the text the system produces

27.4.1 Sources of Token Growth

27.4.1.1 1. Long Context

  • Large prompts—extensive input provided to the system
  • Extensive memory—including too much conversation history
  • Verbose context—unnecessarily detailed information

Impact: Increases processing time and cost

27.4.1.2 2. Verbose Instructions

  • Unnecessary detail—overly specific or redundant instructions
  • Redundant text—saying the same thing multiple ways
  • Excessive examples—too many demonstration examples

Impact: Wastes tokens without improving results

27.4.1.3 3. Large Outputs

  • Long responses—generating more text than needed
  • Unstructured generation—allowing unrestricted output length
  • Verbose responses—wordy or repetitive outputs

Impact: Increases generation time and cost

Poor design leads to token waste. Efficient design minimizes token usage while maintaining quality.

27.5 Latency

Latency is the time it takes for the system to:

  • Receive input—accept user query or data
  • Process—run through the model and tools
  • Produce output—generate and return response

27.5.1 Components of Latency

  • Model processing time—time for LLM to generate response
  • Network delay—time for data to travel between systems
  • Tool execution time—time for external APIs and services
  • System overhead—time for routing, validation, and other operations

27.5.2 Why Latency Matters

  • User experience—slow systems feel unresponsive
  • System responsiveness—real-time applications require fast responses
  • Throughput—slow latency limits how many requests can be handled

Fast systems feel intelligent. Slow systems feel broken, regardless of output quality.

27.6 Cost

Cost is directly tied to token usage and system design:

  • Token usage—primary driver of API costs
  • Number of calls—frequency of API invocations
  • Model type—different models have different pricing

27.6.1 Cost Drivers

  • Large context windows—more tokens per request
  • Frequent API calls—more requests per unit time
  • Complex pipelines—multiple model calls per task
  • Large models—more expensive per token

27.6.2 Implication

Even small inefficiencies:

  • Scale significantly—small per-request costs multiply at scale
  • Increase operational cost—ongoing expenses accumulate
  • Limit feasibility—high costs make systems uneconomical

Cost is a function of design, not just usage. Good design minimizes cost while maintaining quality.

27.7 Trade-offs Between Metrics

System design involves balancing competing requirements:

27.7.1 Token vs Accuracy

  • More tokens → more context → better accuracy
  • But higher cost—increased expense

Trade-off: Determine how much context is needed for required accuracy

27.7.2 Latency vs Quality

  • Faster responses → less processing
  • But potentially lower quality—less time for reasoning

Trade-off: Balance response speed with output quality requirements

27.7.3 Cost vs Reliability

  • More checks → higher cost
  • But better stability—more validation and evaluation

Trade-off: Determine acceptable failure rate vs. cost of prevention

System design requires balancing these factors based on specific requirements and constraints.

27.8 Optimization Strategies

Several strategies can improve performance and reduce cost:

27.8.1 1. Reduce Token Usage

  • Compress context—summarize long histories
  • Remove redundancy—eliminate duplicate information
  • Use summaries—replace long text with concise summaries

Example: Instead of including full conversation history, include a summary of key points.

27.8.2 2. Optimize Instructions

  • Keep concise—use minimal text to convey requirements
  • Avoid unnecessary detail—focus on essential information
  • Remove redundancy—don’t repeat information

Example: Instead of detailed descriptions, use structured formats like JSON.

27.8.3 3. Control Output Length

  • Define limits—specify maximum output length
  • Use structured output—enforce format constraints
  • Request conciseness—ask for brief, focused responses

Example: Specify “under 100 words” or “3 bullet points” to limit output.

27.8.4 4. Efficient Context Management

  • Retrieve only relevant data—select specific information
  • Avoid full history—use summaries instead of full conversations
  • Prioritize recent information—focus on most relevant context

Example: Use retrieval to find only the most relevant documents, not all available.

27.8.5 5. Cache Results

  • Reuse previous outputs—avoid recomputing same inputs
  • Cache intermediate results—store results of expensive operations
  • Implement memoization—remember computation results

Example: Cache summary of a document so it’s not recomputed for subsequent queries.

Efficiency comes from better design, not shortcuts. Optimize structure, not just parameters.

27.9 Architectural Optimization

System architecture significantly impacts performance:

27.9.1 1. Pipeline Design

  • Break tasks into smaller steps—divide complex operations
  • Avoid overloading single calls—distribute work across multiple calls
  • Parallelize independent operations—run multiple steps simultaneously

Example: Instead of one complex instruction, use multiple simpler instructions in sequence.

27.9.2 2. Selective Execution

  • Only run necessary components—skip unneeded processing
  • Conditional execution—run components only when needed
  • Early termination—stop processing when result is clear

Example: Only run expensive retrieval if the query requires external information.

27.9.3 3. Model Selection

  • Use smaller models where possible—not all tasks need largest models
  • Reserve larger models for complex tasks—allocate resources appropriately
  • Cascade models—use smaller models for filtering, larger for final processing

Example: Use a small model to classify queries, then route to appropriate specialist models.

Not all tasks require maximum capability. Match model size to task complexity.

27.10 Measuring Performance

To optimize, you must measure. Performance metrics provide the data needed for improvement.

27.10.1 Key Metrics

  • Tokens per request—measure input and output token usage
  • Latency per step—track time for each operation
  • Cost per task—calculate expense for each operation

27.10.2 Approach

  • Log system behavior—record all operations and metrics
  • Analyze bottlenecks—identify slow or expensive components
  • Iterate design—make changes based on measurement data

Measurement enables optimization. Without metrics, optimization is guesswork.

27.11 Common Mistakes

Several common mistakes undermine performance:

27.11.1 1. Ignoring Token Usage

  • Leads to unexpected costs—bills higher than expected
  • Limits scalability—cannot handle increased load
  • Reduces competitiveness—higher costs than alternatives

Solution: Monitor token usage. Set budgets and alerts.

27.11.2 2. Overloading Context

  • Reduces efficiency—more processing than needed
  • Increases latency—longer response times
  • Increases cost—more tokens processed

Solution: Be selective about context. Include only relevant information.

27.11.3 3. Over-Engineering Pipelines

  • Too many steps—unnecessary complexity
  • Increased cost—more operations than needed
  • Reduced maintainability—harder to understand and modify

Solution: Keep pipelines simple. Only add steps that add value.

27.11.4 4. Optimizing Too Early

  • Premature optimization—optimizing before understanding requirements
  • Reduced flexibility—over-optimized systems are hard to change
  • Wrong focus—optimizing non-critical components

Solution: Measure first, then optimize. Focus on bottlenecks.

27.12 Performance as a System Property

Performance is not determined by a single component. It depends on:

  • Spec design—how requirements are defined
  • Instruction efficiency—how protocols are structured
  • Context management—how information is selected
  • System architecture—how components are organized

Efficiency emerges from the entire system. Optimize the whole, not just parts.

27.13 Final Insight

In the LLM era, performance is not just about speed. It is about:

  • Efficient use of tokens—minimizing waste
  • Balanced latency—responding quickly enough
  • Controlled cost—operating within budget

The best systems are not just correct. They are efficient.

27.14 Closing Statement

As systems scale, performance and cost become critical constraints.

A well-designed system delivers value not just correctly, but efficiently.

27.15 Key Insight

Performance in LLM systems is determined by three core metrics: token usage, latency, and cost. These metrics are tightly coupled—more tokens increase latency and cost. Optimization strategies include reducing token usage, optimizing instructions, controlling output length, efficient context management, and caching results.

27.16 Action Hint

Monitor token usage, latency, and cost. Identify bottlenecks through measurement. Optimize context by including only relevant information. Use structured outputs to control token usage. Select appropriate model size for task complexity. Cache results to avoid recomputation. Balance trade-offs between accuracy, speed, and cost.

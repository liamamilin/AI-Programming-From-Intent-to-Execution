28  构建聊天系统（Building a Chat System）

28.1 聊天系统作为基础架构

聊天系统（chat system）是最常见的 AI 系统形式之一。它们出现在客户服务、教育、个人助理和内部工具中。但构建一个好的、可靠的聊天系统需要的不仅仅是 API 调用。

聊天系统的核心挑战在于：它们是概率性的、上下文依赖的、需要处理各种边界情况（edge cases）。一个简单的 API 包装不足以提供一个生产级的聊天体验。

本章使用最小循环（Minimal Loop）框架逐步构建一个生产质量的聊天系统。

28.2 第 1 步：定义意图（Define Intent）

意图： 构建一个聊天系统，准确回答用户关于文档集的问题。

精化后的目标（Goal Compression）： - 目标：在超过 80% 的情况下提供相关、准确的答案 - 对象：用户关于文档知识库的问题 - 上下文：Web 界面，单个用户会话，文档集最多 1000 个文档 - 约束：答案必须基于提供的文档，在不确定时说明不知道，支持中文和英文 - 成功标准：准确率 > 80%，响应时间 < 3 秒，用户满意度 > 4/5

28.3 第 2 步：创建 Spec

输入（Input）： 用户问题（字符串，最多 500 字符），会话 ID（用于上下文），可选的用户偏好。

输出（Output）： 回答（字符串，最多 1000 字符），来源引用（文档 ID 列表），置信度（高/中/低），后续建议问题（可选）。

约束（Constraints）： - 仅基于提供的内容回答。如果信息不足，明确说明限制 - 回答必须与问题保持相同语言 - 不存储个人数据。不产生幻觉（hallucination） - 引用必须精确匹配信息源

边界情况（Edge Cases）： - 无相关文档：返回”未找到相关信息”并建议其他查询 - 模糊问题：请求澄清而非猜测 - 多语言混合输入：检测主要语言并回复

28.4 第 3 步：编写指令（Write Instruction）

角色：你是一个知识库助手。你根据提供的文档回答用户问题。

任务：回答用户关于知识库的问题。保持简洁和准确。

输入：用户的问题 + 相关文档片段（最多 5 个）

输出要求：
- 基于文档的准确回答
- 每句话引用来源（用 [1][2] 格式）
- 如果信息不足，说明限制

约束：
- 不要编造信息。不要添加文档中不存在的内容
- 保持回答简洁（最多 3 段）
- 与用户使用相同语言回复

边界情况：
- 无相关文档：说"未找到相关信息"
- 模糊问题：请求用户进一步说明

28.5 第 4 步：架构设计（Architecture Design）

主流程（Main Flow）：

用户输入 → 检索相关文档 → 构建上下文 → 生成回答 → 格式化输出 → 返回结果

模块（Modules）： - 检索器（Retriever）：从知识库查找相关文档。支持语义搜索（semantic search）和关键词搜索（keyword search）。混合检索策略以提高召回率。 - 上下文构建器（Context Builder）：将检索到的文档格式化为提示词上下文。管理上下文窗口限制（context window）。优先级排序——最相关的文档优先。 - 生成器（Generator）：基于检索到的文档生成回答。使用定义的指令和约束。生成结构化输出。 - 格式化器（Formatter）：格式化输出为统一结构。添加来源引用。后处理以确保一致性。

记忆（Memory）： - 短期记忆（Short-term）：会话上下文（保留最近 5 轮对话） - 长期记忆（Long-term）：用户偏好（如果可用）

28.6 第 5 步：评估与精化（Evaluate and Refine）

测试用例类型： - 简单问题（事实查询） - 复杂多步骤问题（需要综合多个文档） - 边界情况（无答案的问题、歧义问题、大型上下文、多语言）

评估指标（Evaluation Metrics）： - 准确率（Accuracy）：回答是否正确 - 引用正确性（Citation Accuracy）：引用是否匹配实际文档 - 响应时间（Latency）：从接收到返回的时间 - 用户满意度（User Satisfaction）：通过反馈机制收集

迭代策略（Iteration Strategy）： 1. 运行评估数据集 2. 识别失败模式（定义错误、检索错误、生成错误、边界情况未覆盖） 3. 精化 Spec 和指令 4. 重新评估 5. 重复直到满足质量标准

28.7 部署注意事项（Deployment Considerations）

  • API 封装：将系统封装为 REST API 端点
  • 监控（Monitoring）：追踪令牌使用、延迟、错误率
  • 日志（Logging）：记录每次交互的输入和输出用于调试
  • 成本管理：缓存重复查询、控制输出长度、批处理优化
  • 持续改进：收集用户反馈 → 分析模式 → 精化系统

28.8 小结

构建生产级聊天系统需要的不仅仅是模型调用。它需要一个系统化的方法：从意图（Intent）开始，通过 Spec 和指令（Instruction）精化定义，设计清晰的架构，系统化地评估和迭代。

聊天系统是许多 AI 应用的基础模式。掌握了它的构建方法，你就掌握了构建更复杂 AI 系统的基础。

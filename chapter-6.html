7  规格说明：消除歧义（Spec: Eliminating Ambiguity）

7.1 从意图到精确性

意图定义了我们想要什么——它捕捉了人类的目标和愿望。但仅有意图不足以有效指导系统开发。

意图本质上是不完整的、上下文依赖的、常常是歧义的。

要让系统可靠地工作，意图必须被转化为更精确和可操作的东西。这种转化就是Spec（规格说明）的角色。

Spec 是将模糊的意图变成结构化定义的地方。

7.2 什么是 Spec？（What is a Spec?）

Spec 是一个结构化的、无歧义的系统行为描述。它是人类意图和机器执行之间的桥梁。

一个 Spec 定义：

  • 系统接收什么输入——系统使用的数据和信息
  • 产生什么输出——系统生成的结果和响应
  • 必须遵循什么约束——系统必须尊重的边界和限制
  • 什么条件决定成功——评估系统是否正常工作的标准

与意图不同——意图是人类导向的，可以用自然语言表达——Spec 是：

  • 系统导向的——设计用于机器解释和执行
  • 显式的——不留下任何隐式或假设的东西
  • 可验证的——可以对照客观标准进行测试

7.3 Spec 即约束，而非描述（Spec as Constraint, Not Description）

一个常见且根本性的误解是将 Spec 视为系统的”详细描述”。这并不正确，而且会导致糟糕的规格说明。

Spec 不是以穷尽的细节描述系统做的一切。它是关于定义边界——指定系统必须做什么、不得做什么以及它必须如何行为。

7.3.1 描述 vs 规格说明

描述： “系统应快速、准确地响应用户问题。”

这个描述感觉有帮助，但对系统开发基本无用。“快速”和”准确”是主观术语，无法衡量或测试。

规格说明： “95% 的查询响应时间必须在 2 秒以下。答案在可用时必须引用检索到的文档。当未找到相关文档时，系统必须明确说明这一限制。”

这个规格说明是可操作的。它定义了一组具体、可衡量的约束，可以被实现和测试。

描述解释。规格说明约束。

Spec 的目的不是告诉系统它可能做的一切。而是告诉系统它必须做什么和必须不做什么。

7.4 Spec 的功能（The Function of Spec）

Spec 在系统开发中执行三个关键功能：

7.4.1 1. 消除歧义

自然语言允许多种解释。一个陈述可以对不同的人或系统意味着不同的事情。

Spec 通过以下方式减少这种情况：

  • 定义显式的输入和输出——精确指定什么数据进入和离开
  • 设定具体、可衡量的标准——可测试的标准代替主观术语
  • 使假设显式化——澄清未陈述的期望

7.4.2 2. 启用执行

一旦 Spec 被定义，它可以被转化为系统可以遵循的指令。没有 Spec，指令就没有基础——你不知道系统应该实现什么。

Spec 的可执行性使其与普通文档区分开来。

7.4.3 3. 促进评估

Spec 提供了评估的基础。对照 Spec 定义的标准，你可以判断系统是否工作正常。

没有 Spec，评估是主观和武断的。

7.5 良好 Spec 的特征（Characteristics of a Good Spec）

一个好的 Spec 有几个关键特征：

  • 具体的——避免模糊术语如”快速”或”用户友好”。使用可衡量的定义
  • 可测试的——每个陈述都应该可以被验证为真或假
  • 完整的——涵盖相关的输入、输出、约束和成功标准
  • 一致的——各部分之间没有内部矛盾
  • 最小化的——包含必要的内容，但不再多

一个有用的测试：如果你给两个不同的人（或两个 AI 系统）这个 Spec，他们应该对什么是正确的输出达成一致。

7.6 Spec 不是什么（What Spec is Not）

  • 不是需求文档——需求文档描述问题。Spec 描述解法的结构
  • 不是设计文档——设计文档描述系统如何实现。Spec 描述系统必须做什么
  • 不是代码——Spec 比代码更抽象，但也更结构化

Spec 有自己的类别——一个在人类意图和机器执行之间运行的结构化工件。

7.7 实践：编写一个 Spec（Practice: Writing a Spec）

第一步：从意图开始

意图：“构建一个客户服务聊天机器人”。

第二步：定义输入

用户通过 Web 界面提交自然语言问题。输入长度最多 2000 字符。系统可以使用可选的用户上下文。

第三步：定义输出

系统返回 50-500 字符的自然语言回复。回复可以是答案、澄清问题或承认无法回答。系统在有数据时提供来源引用。

第四步：定义约束

回复时间 < 2 秒（95%）。系统不得编造信息。系统在无法回答时必须明确说明。系统必须使用与用户相同的语言回复。

第五步：定义成功标准

用户反馈满意度 > 80%。首次回复解决率 > 60%。每周 7 天、每天 24 小时可用。

7.8 Spec 作为桥梁（Spec as Bridge）

Spec 是意图和指令之间的桥梁。它将人类的”想要”转化为系统的”做”。

没有 Spec，指令就没有基础，评估就没有标准。Spec 是将编程从模糊想法转变为精确结果的核心工件。

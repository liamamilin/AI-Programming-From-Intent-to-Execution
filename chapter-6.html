7  Spec: Eliminating Ambiguity

7.1 From Intent to Precision

Intent defines what we want—it captures the human goal and aspiration. But intent alone is not enough to guide system development effectively.

Intent, by its nature, is:

  • Incomplete—it doesn’t specify all necessary details
  • Context-dependent—it relies on unstated assumptions and background knowledge
  • Often ambiguous—it can be interpreted in multiple ways

To make a system work reliably, intent must be transformed into something more precise and actionable. This transformation is the role of Spec (Specification).

Spec is where vague intention becomes structured definition.

7.2 What is a Spec?

A Spec is a structured, unambiguous description of a system’s behavior. It is the bridge between human intent and machine execution.

A spec defines:

  • What inputs the system receives—the data and information the system works with
  • What outputs it produces—the results and responses the system generates
  • What constraints it must follow—the boundaries and limitations the system must respect
  • What conditions determine success—the criteria for evaluating whether the system works correctly

Unlike intent, which is human-oriented and can be expressed in natural language, spec is:

  • System-oriented—designed for machine interpretation and execution
  • Explicit—leaves nothing implicit or assumed
  • Verifiable—can be tested against objective criteria

7.3 Spec as Constraint, Not Description

A common and fundamental misunderstanding is to treat a spec as a “detailed description” of a system. This is incorrect and leads to poor specifications.

A spec is not about describing everything a system does in exhaustive detail. It is about defining boundaries—specifying what the system must do, must not do, and how it must behave.

7.3.1 Description vs Specification

Consider this example:

Description: > “The system should respond quickly and accurately to user questions.”

This description feels helpful but is fundamentally useless for system development. “Quickly” and “accurately” are subjective terms that cannot be measured or tested.

Specification: > “Response time must be under 2 seconds for 95% of queries. Answers must reference retrieved documents when available. When no relevant documents are found, the system must explicitly state this limitation.”

This specification is actionable. It defines concrete, measurable constraints that can be implemented and tested.

Description explains. Specification constrains.

The purpose of a spec is not to tell the system everything it might do. It is to tell the system what it must do and what it must not do.

7.4 The Function of Spec

Spec performs three critical functions in system development:

7.4.1 1. Remove Ambiguity

Natural language allows multiple interpretations. A single statement can mean different things to different people or systems.

Spec reduces this by:

  • Defining explicit inputs and outputs—specifying exactly what data goes in and comes out
  • Clarifying edge cases—determining how the system should handle unusual situations
  • Making assumptions visible—explicitly stating what is being assumed about context or environment

Without spec, different implementations will interpret the same intent in different ways, leading to inconsistent behavior.

7.4.2 2. Align Humans and Systems

Spec serves as a shared contract between different stakeholders:

  • Humans—designers, developers, users, and stakeholders
  • Systems—LLMs, agents, and execution engines

It ensures:

  • Everyone is operating under the same definition
  • There is no hidden interpretation gap between human intent and machine execution
  • Success criteria are agreed upon upfront

Without this alignment, systems may behave in ways that surprise their human counterparts, and humans may have unrealistic expectations of what systems can achieve.

7.4.3 3. Enable Execution

Without spec:

  • Systems must guess at what is intended
  • Outputs vary widely between runs
  • Behavior is unpredictable and unreliable

With spec:

  • Systems have clear constraints to operate within
  • Outputs become consistent and predictable
  • Behavior becomes controllable and verifiable

Spec provides the guidance necessary for automated execution to produce reliable results.

7.5 Why Spec is the Core Layer

In the minimal loop:

Intent → Spec → Instruction → Execution → Evaluation

Spec sits at the center of the loop. This is not accidental—it reflects spec’s critical role.

7.5.1 It connects both sides

  • Upstream: human intent and understanding
  • Downstream: machine execution and behavior

Spec is the translation layer that converts human thinking into machine-understandable form.

7.5.2 It transforms meaning

  • From semantic → structural: converting meaning into formal structure
  • From vague → explicit: making implicit assumptions clear
  • From idea → system: transforming abstract concepts into concrete definitions

This transformation is essential for bridging the gap between human cognition and machine execution.

Spec is the bridge between thinking and execution.

Without spec, there is no reliable way to convert human intent into machine behavior.

7.6 The Cost of Missing Spec

When spec is weak or absent from the development process, several problems emerge:

7.6.1 1. Systems behave unpredictably

  • Different outputs for similar inputs
  • Inconsistent behavior across different runs
  • Unexpected responses in edge cases

The system becomes unreliable and difficult to trust.

7.6.2 2. Iteration becomes inefficient

  • Changes are trial-and-error rather than systematic
  • Improvements are not cumulative—each iteration starts from scratch
  • Progress is slow and directionless

Without clear specs, you don’t know what to change or how to measure improvement.

7.6.3 3. Errors are misdiagnosed

Developers often blame:

  • The model—“This LLM isn’t smart enough”
  • The tool—“This framework has bugs”
  • The framework—“This approach doesn’t work”

But the real issue is often:

The system was never clearly defined.

Blaming tools and technologies when the real problem is poor spec leads to wasted effort and frustration.

7.7 Spec vs Code

Traditionally, code served as the primary definition of a system’s behavior. The code was the specification—if you wanted to know what a system did, you read the code.

Now, that role is shifting. Spec is becoming the primary definition, with code serving as one possible realization of that spec.

7.7.1 Before

  • Code = behavior
  • Logic embedded in implementation details
  • Understanding required reading code

7.7.2 Now

  • Spec = behavior
  • Code = one possible implementation of the spec
  • Understanding requires reading the spec

Code implements. Spec defines.

This shift is profound. It means that behavior is no longer determined by implementation details, but by explicit specification. Multiple different code implementations could satisfy the same spec, and changing the implementation doesn’t necessarily change the behavior.

7.8 Spec as a First-Class Artifact

In the LLM era, spec becomes:

  • A primary artifact—as important as code, perhaps more so
  • A reusable asset—can be shared, improved, and applied across projects
  • A unit of iteration—the thing we refine to improve systems

Instead of focusing on:

  • Code repositories—managing and organizing code

We begin to focus on:

  • Spec repositories—managing and organizing specifications

This shift requires new tools, new practices, and new ways of thinking about what constitutes the “source code” of a system.

7.9 Spec Enables Iteration

Iteration in traditional programming often focuses on code changes—finding and fixing bugs, optimizing performance, refactoring for clarity.

In the new paradigm:

  • Iteration focuses on refining the spec

The process becomes:

  • Define spec
  • Execute
  • Evaluate
  • Adjust spec

Improvement comes from better definition, not more code.

This is a more efficient form of iteration because spec is at a higher level of abstraction than code. Small changes in spec can lead to large improvements in system behavior.

7.10 Spec and Control

One concern that often arises with this approach is the loss of control:

“If the system executes automatically based on a spec, do we lose control?”

In reality, control shifts rather than disappears:

  • From controlling steps—specifying every operation and decision
  • To controlling constraints—defining the boundaries within which the system operates

You no longer dictate every action the system takes. Instead, you define the space within which actions occur. You control the boundaries, not the path.

This is analogous to herding sheep versus directing individual sheep. The shepherd doesn’t control each sheep’s movement directly but defines boundaries that keep the flock moving in the right direction.

7.11 Common Mistakes

7.11.1 1. Treating Spec as Optional

Skipping spec or treating it as a mere formality leads to:

  • Unstable systems with unpredictable behavior
  • Poor reproducibility—different runs produce different results
  • Difficulty debugging and improving the system

Spec is not optional. It is essential.

7.11.2 2. Over-specifying

Trying to define every detail, every edge case, every possible behavior:

  • Reduces flexibility and adaptability
  • Conflicts with the probabilistic nature of LLM systems
  • Creates maintenance burden

Good spec finds the right balance between clarity and flexibility.

7.11.3 3. Under-specifying

Leaving too much implicit, assuming the system will figure it out:

  • Leads to ambiguity and multiple interpretations
  • Produces inconsistent and unpredictable results
  • Makes evaluation impossible

You must be explicit about what matters.

Good spec balances clarity and flexibility. It specifies what must be true without dictating how it must be achieved.

7.12 Final Insight

Spec is not an intermediate step or a preliminary activity. It is the core of the system. It determines:

  • What the system does—the behavior it exhibits
  • How it behaves—the constraints it operates within
  • Whether it succeeds—whether it meets the intended goals

Without a good spec, no amount of clever code or powerful tools can produce a reliable system. With a good spec, implementation becomes straightforward and almost mechanical.

7.13 Closing Statement

In the LLM era, the most important artifact is no longer code. It is specification.

If intent defines what you want, spec determines whether you can actually get it.

The ability to write good specs is now a core programming skill—perhaps the most important skill.

7.14 Key Insight

Spec transforms vague intent into structured, unambiguous definition. It functions as a contract between humans and systems, removing ambiguity and enabling reliable execution.

7.15 Action Hint

Practice writing specs for simple systems. Use this structure: Input, Output, Constraints, Success Criteria. Make each element explicit and testable. Review your specs for ambiguity—any vagueness will propagate through the entire system.

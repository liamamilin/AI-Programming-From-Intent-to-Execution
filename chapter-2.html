3  The New Bottleneck

3.1 Where Does Difficulty Come From?

In any system of production, there is always a limiting factor—the bottleneck.

In traditional programming, the bottleneck was clear:

  • Writing correct code

  • Managing complexity

  • Handling edge cases

  • Debugging failures

In other words:

Execution was the bottleneck.

3.2 Execution Has Been Outsourced

With the rise of LLMs and AI systems, execution is no longer the primary constraint.

Today, a developer can:

  • Generate code from natural language

  • Scaffold entire applications in minutes

  • Reuse patterns automatically

  • Delegate tasks to agents

What once required hours or days can now be done in seconds.

3.2.1 What “outsourcing execution” really means

It does not mean execution disappears.

It means:

Execution is no longer performed directly by humans.

Instead, it is handled by:

  • Language models

  • Agent systems

  • Toolchains and APIs

The developer no longer “does the work” step by step.

They define the work, and the system performs it.

3.3 The Illusion of Productivity

At first glance, this seems like a pure productivity gain.

  • Faster development

  • Less manual effort

  • More output

However, a new problem quickly emerges:

The system produces exactly what you ask for—no more, no less.

If the input is vague, incomplete, or inconsistent:

  • The output will reflect that

  • Errors will shift upstream

  • Failures become harder to diagnose

3.4 The New Bottleneck: Definition

As execution becomes easier, another constraint becomes visible:

  • What exactly are we building?

  • What are the requirements?

  • What does “correct” mean?

These questions were always present, but previously hidden by implementation effort.

Now they are exposed.

Definition becomes the bottleneck.

3.4.1 Symptoms of a Definition Bottleneck

You are likely facing a definition problem if:

  • You keep rewriting prompts but results remain unstable

  • The system “almost works” but never fully satisfies the goal

  • Different runs produce inconsistent outputs

  • You cannot clearly explain what success looks like

These are not execution failures.

They are definition failures.

3.5 From Code Errors to Definition Errors

The nature of errors has fundamentally changed.

3.5.1 Before (Execution Errors)

  • Syntax errors

  • Runtime exceptions

  • Logical bugs

  • Performance issues

These errors are:

  • Localized

  • Deterministic

  • Debuggable through code inspection

3.5.2 Now (Definition Errors)

  • Ambiguous goals

  • Missing constraints

  • Undefined edge cases

  • Weak evaluation criteria

These errors are:

  • Global

  • Context-dependent

  • Difficult to trace

The system is not “wrong”—it is following an incomplete definition.

3.6 Productivity Reframed

We can now redefine productivity in programming.

3.6.1 Traditional Model

Productivity = Implementation Skill × Time

Improvement comes from:

  • Writing faster

  • Debugging better

  • Optimizing execution

3.6.2 LLM Era Model

Productivity = Definition Quality × Execution Capability

Where:

  • Execution Capability is largely provided by AI

  • Definition Quality becomes the primary variable

3.6.3 Implication

Even with powerful AI:

  • Poor definition → poor results

  • Clear definition → high leverage

AI amplifies your clarity, not your intent.

3.7 The Asymmetry of Leverage

This new model introduces an asymmetry:

  • A small improvement in definition can produce large gains in output

  • A small ambiguity can cause large failures

This makes definition a high-leverage activity.

3.7.1 Example

Two developers use the same AI system:

  • Developer A provides vague instructions → inconsistent results

  • Developer B provides structured definitions → reliable system

The difference is not in tools.

It is in definition quality.

3.8 The Hidden Work

In traditional programming, most effort is visible:

  • Writing code

  • Running tests

  • Fixing bugs

In the LLM paradigm, much of the critical work becomes invisible:

  • Clarifying intent

  • Structuring requirements

  • Defining constraints

  • Designing evaluation

This work is often underestimated because:

  • It produces no immediate output

  • It feels like “thinking,” not “building”

Yet it is now the dominant factor in success.

3.9 A Change in Mindset

To operate effectively, one must shift mindset:

From:

  • “How do I implement this?”

To:

  • “Have I defined this clearly enough for a system to execute?”

3.10 Final Insight

The bottleneck in programming has moved.

Not from one tool to another,
but from one layer of thinking to another.

Execution has been outsourced.
Definition has become the constraint.

3.11 Closing Statement

In the LLM era, the quality of what you build is no longer limited by your ability to execute.

It is limited by your ability to define.

The better you define, the more powerful the system becomes.

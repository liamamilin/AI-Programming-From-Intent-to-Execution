3  新的瓶颈：定义（The New Bottleneck: Definition）

3.1 难度从何而来？

在任何生产系统中，总有一个限制因素——瓶颈。这是系统中约束整体绩效的点，是额外投入产生最大回报的地方。理解瓶颈在哪里对于有效的系统设计和优化至关重要。

在传统软件开发中，瓶颈是清晰且普遍被认可的：

  • 编写正确的代码来实现所需的逻辑
  • 管理日益增长的代码库的复杂性
  • 全面处理边界情况（edge cases）和错误条件
  • 调试失败并追踪复杂系统中的执行路径

换句话说：

执行（execution）是瓶颈。

能构建什么的限制因素不是想法的清晰度，而是正确高效地实现它的能力。更好的工具、更快的计算机和改进的方法论都聚焦于缓解这个执行瓶颈。

3.2 执行已被外包（Execution Has Been Outsourced）

随着大语言模型的崛起，执行不再是软件开发中的首要约束。这代表了编程经济学的根本性转变。

今天，配备了现代 AI 工具的开发者可以：

  • 从自然语言描述中在几秒内生成可运行的代码
  • 在几分钟内搭建完整的应用结构（原来需要数小时）
  • 在项目间自动复用模式和最佳实践
  • 将常规任务委托给专门智能体可靠地处理

曾经需要数小时或数天的手动工作，现在可以在数秒或数分钟内完成。执行的障碍已经大幅下降。

3.2.1 “外包执行”到底意味着什么

执行不再由人类直接执行。而是由语言模型、智能体系统、工具链和 API 来处理。

开发者不再一步步”做工作”——逐行写代码、手动处理每个边界情况。相反，他们以更高层面定义工作，系统执行具体细节。

这类似于从手工制作家具到管理自动化工厂的转变。曾经亲自锯木板、钉钉子的木匠，现在设计产品并管理生产它的机器。执行已被外包，但对设计和质量保证的需求仍然存在。

3.3 生产力的幻觉（The Illusion of Productivity）

乍看之下，这种转变似乎是纯粹的生产力提升——一种让一切更快更简单的直接改进。开发者可以更快地生成代码、更快速地迭代、更广泛地原型设计。

然而，一个新问题迅速浮现，不太明显但同样重要：

系统精确地产生你所要求的——不多也不少。

这看似直白的陈述有着深远的影响。如果系统的输入是模糊的、不完整的、不一致的或歧义的，那么输出将忠实地反映这些品质。错误将从执行失败上游转移到定义失败。失败的诊断变得更困难，因为它们不是代码中的 bug，而是理解中的缺陷。

生产力幻觉来自于假设更快的执行自动产生更好的结果。实际上，对定义不良的任务的更快执行，只是更快地产生定义不良的结果。

3.4 新的瓶颈：定义（The New Bottleneck: Definition）

随着执行变得更容易和更自动化，另一个约束变得可见——它之前被实现的工作量所隐藏。

我们现在面临的问题一直存在但很少被审视：

  • 我们究竟在构建什么？
  • 完整的需求是什么？
  • 在这个上下文中，“正确”意味着什么？
  • 系统必须遵守什么约束？
  • 我们怎么知道我们成功了？

这些问题一直很重要，但它们常常通过迭代实现的过程被隐式地处理。随着开发者编写代码并遇到问题，他们优化了对构建内容的理解。实现的反馈循环充当了一个发现机制。

现在执行被外包了，这种隐式的发现过程被打乱了。我们不能再依赖编码行为来澄清我们的思考。我们必须在执行之前就保持清晰。

定义成为瓶颈。

3.4.1 定义瓶颈的症状

  • 提示不稳定：你不断重写提示或规格，但结果仍然不一致
  • “差不多”问题：系统”差不多能工作”但从未完全满足目标
  • 输出不一致：类似输入在不同运行中产生显著不同的输出
  • 无法阐述成功：你不能清晰解释成功是什么样或如何衡量
  • 无尽调整：你发现自己无休止地做小调整而没有有意义的改进

这些不是传统意义上的执行失败。系统没有”坏掉”或”有 bug”——它忠实地遵循了你的定义。它们是定义失败——系统精确地做了你所要求的，但你所要求的并不是你真正想要的。

3.5 从代码错误到定义错误（From Code Errors to Definition Errors）

软件开发中错误的本质已经彻底改变。

3.5.1 以前（执行错误）

在传统编程中，错误主要是技术性的和局部的：语法错误、运行时异常、逻辑 bug、性能问题。这些错误共有的特征是局部化、确定性、可通过代码检查调试。

3.5.2 现在（定义错误）

在基于 LLM 的系统中，错误具有不同的性质：模糊的目标、缺失的约束、未定义的边界情况、薄弱的评估标准。这些新类型的错误是全局的、上下文相关的、难以追踪的。

系统不是”错”了——它是在遵循一个不完整的定义。

区别微妙但关键。在传统调试中，你问”代码哪里错了？“在新范式中，你问”定义哪里不完整？”

3.6 生产力被重新定义（Productivity Reframed）

3.6.1 传统模型

生产力 = 实现技能 × 时间。改进来自写更快、调试更好、优化执行。

3.6.2 LLM 时代模型

生产力 = 定义质量 × 执行能力。执行能力由 AI 提供且日益商品化，定义质量成为区分结果的主要变量。

AI 放大的是你的清晰度，而不是你的意图。

这是 LLM 时代也许最重要的洞见。你带来的价值不是有好的想法——每个人都有好的想法。价值在于能够以足够的精确度定义这些想法，使系统可靠地执行它们。

3.7 杠杆的不对称性（The Asymmetry of Leverage）

这个新模型引入了一个重要的不对称性：

  • 定义的小改进可以产生输出质量和一致性的大幅提升
  • 一个小的歧义可能导致大的失败和不可预测的行为

这使得定义成为一个高杠杆活动——清晰度的小额投资在结果质量上产生大额回报。

3.8 隐形工作（The Hidden Work）

在传统编程中，大部分工作可见且可衡量：写代码、运行测试、修复 bug。

在 LLM 范式中，关键工作变得不可见：澄清意图、结构化需求、定义约束、设计评估。这项工作通常不产生即时可见的输出。它发生在思考、对话、文档和规划中。

然而，这种隐形工作现在是决定成功的主要因素。没有它，可见工作——快速的代码生成、快速的原型、轻松的迭代——产生的是噪音而不是信号。

3.9 思维方式的转变（A Change in Mindset）

从： “我如何实现这个？” 到： “我是否已经定义得足够清晰，让系统可以执行？”

在开始任何工作之前，有效的开发者在 LLM 时代会问：

  • 我到底想要它做什么？
  • 这个问题的边界是什么？
  • 成功是什么样子的？
  • 我怎么知道我已经成功了？
  • 我没有说出来的、需要明确的是什么？

只有这些问题可以清晰回答时，才应该继续执行。

执行已被外包。定义已成为约束。

3  Where Does Difficulty Come From?

In any system of production, there is always a limiting factor—the bottleneck. This is the point in the system that constrains overall performance, the place where additional investment yields the greatest return. Understanding where the bottleneck lies is essential for effective system design and optimization.

In traditional software development, the bottleneck was clear and universally recognized:

  • Writing correct code that implements the required logic
  • Managing the complexity of increasingly large codebases
  • Handling edge cases and error conditions comprehensively
  • Debugging failures and tracing execution paths through complex systems

In other words:

Execution was the bottleneck.

The primary limitation on what could be built was not the clarity of the idea, but the ability to implement it correctly and efficiently. Better tools, faster computers, and improved methodologies all focused on alleviating this execution bottleneck.

3.1 Execution Has Been Outsourced

With the rise of Large Language Models and AI systems, execution is no longer the primary constraint in software development. This represents a fundamental shift in the economics of programming.

Today, a developer equipped with modern AI tools can:

  • Generate working code from natural language descriptions in seconds
  • Scaffold entire application structures in minutes that would have taken hours
  • Reuse patterns and best practices automatically across projects
  • Delegate routine tasks to specialized agents that handle them reliably

What once required hours or days of manual effort can now be accomplished in seconds or minutes. The barrier to execution has dropped dramatically.

3.1.1 What “outsourcing execution” really means

It is important to be precise about what this means. We are not saying that execution has disappeared or become unimportant. Systems still need to execute logic, code still needs to run, and implementation details still matter.

What has changed is who performs the execution:

Execution is no longer performed directly by humans.

Instead, it is handled by:

  • Language models that generate code from specifications
  • Agent systems that coordinate multi-step workflows
  • Toolchains and APIs that encapsulate complex functionality

The developer no longer “does the work” step by step, writing each line of code and handling each edge case manually. Instead, they define the work at a higher level, and the system performs the detailed execution.

This is analogous to the shift from hand-crafting furniture to managing an automated factory. The carpenter who once cut each board and drove each nail now designs the product and manages the machinery that produces it. The execution has been outsourced, but the need for design and quality assurance remains.

3.2 The Illusion of Productivity

At first glance, this shift seems like a pure productivity gain—a straightforward improvement that makes everything faster and easier. Developers can generate code more quickly, iterate more rapidly, and prototype more extensively.

  • Faster development cycles as code generation accelerates
  • Less manual effort spent on routine implementation tasks
  • More output in less time, seemingly unlimited productivity

However, a new problem quickly emerges that is less immediately apparent but equally significant:

The system produces exactly what you ask for—no more, no less.

This seemingly straightforward statement has profound implications. If the input to the system is vague, incomplete, inconsistent, or ambiguous, then:

  • The output will faithfully reflect those qualities
  • Errors will shift from implementation failures upstream to definition failures
  • Failures become harder to diagnose because they are not bugs in code but bugs in understanding

The illusion of productivity comes from assuming that faster execution automatically produces better results. In reality, faster execution of poorly defined tasks produces poorly defined results more quickly.

3.3 The New Bottleneck: Definition

As execution becomes easier and more automated, another constraint becomes visible that was previously hidden by the effort of implementation. This constraint emerges precisely because execution is no longer the limiting factor.

We now face questions that were always present but rarely examined:

  • What exactly are we building?
  • What are the complete requirements?
  • What does “correct” mean in this context?
  • What constraints must the system respect?
  • How will we know when we have succeeded?

These questions were always important, but they were often addressed implicitly through the iterative process of implementation. As developers wrote code and encountered problems, they refined their understanding of what they were building. The feedback loop of implementation served as a discovery mechanism.

Now that execution is outsourced, this implicit discovery process is disrupted. We can no longer rely on the act of coding to clarify our thinking. We must be clear before we execute.

Definition becomes the bottleneck.

3.3.1 Symptoms of a Definition Bottleneck

How can you tell if you are facing a definition problem rather than an execution problem? Look for these common symptoms:

  • Prompt instability: You keep rewriting prompts or specifications, but results remain inconsistent and unpredictable
  • The “almost” problem: The system “almost works” but never fully satisfies the intended goal—there’s always something slightly off
  • Output inconsistency: Different runs with similar inputs produce significantly different outputs
  • Inability to articulate success: You cannot clearly explain what success looks like or how to measure it
  • Endless tweaking: You find yourself making endless small adjustments without meaningful improvement

These are not execution failures in the traditional sense. The system is not “broken” or “buggy”—it is following your definition faithfully.

They are definition failures—the system is doing exactly what you asked for, but what you asked for was not what you actually wanted.

3.4 From Code Errors to Definition Errors

The nature of errors in software development has fundamentally changed. Understanding this change is crucial for effective debugging and improvement in the LLM era.

3.4.1 Before (Execution Errors)

In traditional programming, errors were primarily technical and localized:

  • Syntax errors—mistakes in the grammar of the programming language
  • Runtime exceptions—errors that occur during execution, such as null pointer exceptions
  • Logical bugs—incorrect implementations of intended logic
  • Performance issues—inefficient code that doesn’t meet performance requirements

These errors shared important characteristics:

  • They were localized—you could often trace them to specific lines of code or functions
  • They were deterministic—the same input consistently produced the same error
  • They were debuggable through code inspection—you could understand the problem by examining the code

3.4.2 Now (Definition Errors)

In LLM-based systems, errors have a different character:

  • Ambiguous goals—the specification doesn’t clearly define what should be achieved
  • Missing constraints—important boundaries or limitations are not specified
  • Undefined edge cases—the system doesn’t know how to handle unusual inputs
  • Weak evaluation criteria—there’s no clear way to measure success

These new types of errors are fundamentally different:

  • They are global—problems often stem from the overall definition rather than specific implementation details
  • They are context-dependent—the same definition may work in some contexts but fail in others
  • They are difficult to trace—the error is not in the code but in the thinking that produced the specification

The system is not “wrong”—it is following an incomplete definition.

The difference is subtle but crucial. In traditional debugging, you ask “Where is the code wrong?” In the new paradigm, you ask “Where is the definition incomplete?”

3.5 Productivity Reframed

The shift from execution bottleneck to definition bottleneck requires us to reframe our understanding of productivity in programming. The formula for productivity has changed.

3.5.1 Traditional Model

In the traditional model, productivity could be expressed as:

Productivity = Implementation Skill × Time

Improvement came from:

  • Writing faster—increasing typing speed and code fluency
  • Debugging better—finding and fixing errors more efficiently
  • Optimizing execution—making code run faster and use fewer resources

The focus was on doing more in less time through technical skill and efficiency.

3.5.2 LLM Era Model

In the new model, productivity looks different:

Productivity = Definition Quality × Execution Capability

Where:

  • Execution Capability is largely provided by AI systems and is increasingly commoditized
  • Definition Quality becomes the primary variable that differentiates outcomes

This is a profound shift. It means that:

  • Two developers using identical AI tools can produce vastly different results
  • The difference is not in technical implementation skill but in clarity of thinking
  • Improving productivity requires improving definition, not learning more syntax

3.5.3 Implication

Even with the most powerful AI systems available:

  • Poor definition → poor results: No amount of computational power can compensate for unclear requirements
  • Clear definition → high leverage: A well-defined problem enables exponential amplification through AI execution

AI amplifies your clarity, not your intent.

This is perhaps the most important insight of the LLM era. The value you bring is not in having good ideas—everyone has good ideas. The value is in the ability to define those ideas with sufficient precision that a system can execute them reliably.

3.6 The Asymmetry of Leverage

This new model introduces an important asymmetry that has significant implications for how we approach work:

  • A small improvement in definition can produce large gains in output quality and consistency
  • A small ambiguity can cause large failures and unpredictable behavior

This makes definition a high-leverage activity—small investments in clarity yield large returns in outcome quality.

3.6.1 Example

Consider two developers using the same AI system to build a document summarization tool:

  • Developer A provides vague instructions: “Summarize this document”
    • Results vary widely between runs
    • Sometimes summaries are too long, sometimes too brief
    • Key information is sometimes missed
    • The tool is unreliable and frustrating to use
  • Developer B provides structured definitions:
    • “Extract the 3-5 most important points”
    • “Each point should be 10-15 words”
    • “Prioritize actionable insights”
    • “Exclude introductory and concluding material”
    • Results are consistent and predictable
    • The tool reliably produces useful summaries

The difference is not in the tools used—both have access to the same AI capabilities. The difference is in definition quality.

3.7 The Hidden Work

In traditional programming, most of the effort involved in creating software was visible and measurable:

  • Writing code—the physical act of creating the implementation
  • Running tests—verifying that code works as intended
  • Fixing bugs—tracing and correcting errors

These activities produce tangible outputs that can be counted, measured, and tracked. They fit easily into project management frameworks and productivity metrics.

In the LLM paradigm, much of the critical work becomes invisible:

  • Clarifying intent—making vague ideas precise and actionable
  • Structuring requirements—organizing chaotic thoughts into coherent specifications
  • Defining constraints—articulating boundaries and limitations explicitly
  • Designing evaluation—creating criteria for measuring success

This work often produces no immediate visible output. It happens in thinking, in conversations, in documentation, and in planning. It feels like “thinking” rather than “building,” and as a result, it is often underestimated and under-resourced.

Yet this invisible work is now the dominant factor in determining success. Without it, the visible work—the rapid code generation, the quick prototyping, the easy iteration—produces noise rather than signal.

3.8 A Change in Mindset

To operate effectively in this new paradigm, one must fundamentally shift mindset. The default mode of thinking for experienced developers must be updated.

From: - “How do I implement this?”

To: - “Have I defined this clearly enough for a system to execute?”

This seems like a small shift, but it represents a fundamental change in approach. The first question focuses on implementation technique. The second question focuses on definition quality.

Before starting any work, the effective developer in the LLM era asks:

  • What exactly do I want this to do?
  • What are the boundaries of this problem?
  • What does success look like?
  • How will I know when I’ve achieved it?
  • What am I not saying that needs to be explicit?

Only when these questions can be answered clearly should the developer proceed to execution.

3.9 Final Insight

The bottleneck in programming has moved. Not from one tool to another, and not from one technique to another, but from one layer of thinking to another.

Execution has been outsourced.
Definition has become the constraint.

This shift represents both a challenge and an opportunity. The challenge is that we must develop new skills and new ways of thinking. The opportunity is that by mastering definition, we can achieve levels of productivity and capability that were previously impossible.

3.10 Closing Statement

In the LLM era, the quality of what you build is no longer limited by your ability to execute code. It is limited by your ability to define problems, structures, and outcomes with clarity and precision.

The better you define, the more powerful the system becomes.

3.11 Key Insight

The bottleneck has shifted from execution to definition. AI provides unlimited execution capability; the limiting factor is your ability to define clearly what should be built.

3.12 Action Hint

Before your next AI-assisted coding session, write down the definition first. Use a structured format: goal, inputs, outputs, constraints, and success criteria. Only after you can explain what you want clearly should you let the AI execute.

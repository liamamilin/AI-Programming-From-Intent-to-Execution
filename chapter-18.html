19  From Spec to Instruction

19.1 The Missing Link

By now, we have defined two critical elements of programming in the LLM era:

  • Intent defines what we want—the human goal and aspiration
  • Spec defines what it means—the structured, precise specification of behavior

But systems do not execute specs directly. A spec, no matter how well-written, cannot be fed directly into an LLM or agent system and expected to produce results.

There is a necessary transformation that must occur:

Spec must be translated into Instruction

This translation is the bridge between definition and execution—a critical step that enables systems to actually do work.

19.2 Why Spec Cannot Be Executed Directly

A spec defines what a system should do, but it does not define how to invoke the system or what exact form the execution should take.

A spec defines:

  • Structure—the organization of the system
  • Constraints—the boundaries the system must respect
  • Success conditions—how to measure whether it works

But it does not define:

  • How to invoke the system—what calls to make, in what order
  • What exact inputs to pass—the specific format and content of inputs
  • What output format to enforce at runtime—how to structure the results

Spec is:

  • Declarative—it describes what should exist

Instruction is:

  • Operational—it specifies how to make it exist

Spec defines the system. Instruction runs the system.

19.3 What Does “Compilation” Mean?

The transformation from spec to instruction can be understood through the concept of compilation.

19.3.1 Analogy

Think of it like traditional programming:

  • Spec → high-level definition, like source code in Python or Java
  • Instruction → executable form, like machine code or bytecode

19.3.2 Key Idea

Compilation does not change meaning—it changes form:

Spec (definition) → Instruction (execution protocol)

The transformation preserves the intent and requirements of the spec while making them executable by the system.

19.4 The Transformation Process

Converting a spec into an instruction involves several steps, each of which transforms the specification into a more concrete, operational form.

19.4.1 Step 1: Extract Task

From the spec, identify the core action or operation that needs to be performed.

Example: - Spec: “Document summarization system” - Task: “Summarize the input text”

19.4.2 Step 2: Map Input

Define how input will be passed to the system. This includes format, structure, and validation.

Example: - Spec input: “Text document” - Instruction input: {text: string, max_length: 10000}

19.4.3 Step 3: Define Output Structure

Convert output expectations into a structured format that the system must follow.

Example: - Spec output: “Summary and key points” - Instruction output: {summary: string, key_points: Array<string>}

19.4.4 Step 4: Translate Constraints

Turn constraints into enforceable rules that the system must follow during execution.

Example: - Spec constraint: “Under 100 words” - Instruction constraint: "summary_length < 100"

19.4.5 Step 5: Encode Evaluation Signals (Optional)

Add guidance for correctness or hints that help the system produce better results.

Example: - Spec: “Preserve main ideas” - Instruction: “Prioritize information that appears most frequently in the source”

Spec → Task + Input + Output + Constraints → Instruction

19.5 Example Transformation

Let’s walk through a complete example of transforming a spec into an instruction.

19.5.1 Spec (Simplified)

  • Input: Text document
  • Output: Summary
  • Constraints: Max 100 words
  • Success: Captures key points

19.5.2 Instruction

  • Task: Summarize the input text
  • Input: {text: string}
  • Output: {summary: string}
  • Constraints:
    • Length: max 100 words
    • Content: include key points
    • Style: concise and clear

The meaning is preserved—the system still needs to summarize text—but the form becomes executable. The instruction provides the specific protocol the system will follow.

19.6 What Changes During Transformation

The transformation from spec to instruction involves several important changes:

19.6.1 From Abstract to Concrete

Spec concepts become specific, operational requirements.

Example: - Spec: “Be concise” - Instruction: “Limit output to 100 words”

19.6.2 From Implicit to Explicit

Assumptions and general guidance become specific requirements.

Example: - Spec: “Use relevant information” - Instruction: “Use only the provided input text, do not add external knowledge”

19.6.3 From Flexible to Operational

General guidelines become specific rules for execution.

Example: - Spec: “Capture main ideas” - Instruction: “Extract 3-5 key points, each under 20 words, prioritizing frequently mentioned concepts”

The transformation reduces ambiguity at execution time, making behavior more predictable.

19.7 The Role of Instruction in the System

Instruction acts as the interface between definition and execution—the point where abstract concepts become concrete operations.

19.7.1 It ensures:

  • The system knows what to do—clear task definition
  • The system knows how to respond—specific input/output formats
  • The system produces usable outputs—structured results that downstream systems can use

Without instruction:

  • Spec remains unused—a good idea that never becomes reality
  • Execution becomes inconsistent—different systems interpret the spec differently
  • Integration becomes difficult—no standard interface for interaction

19.8 Automation of the Transformation

In advanced systems, the transformation from spec to instruction can be automated, reducing manual effort and ensuring consistency.

19.8.1 Spec → Instruction Agent

An agent that:

  • Takes spec as input
  • Generates instruction automatically
  • Follows defined patterns and best practices

19.8.2 Benefits

  • Faster system creation—no manual translation required
  • Consistent instruction format—standardized across systems
  • Reduced manual effort—less human intervention needed

19.8.3 Risks

  • Incorrect mapping—automated translation may miss nuances
  • Loss of nuance—subtle requirements may be oversimplified
  • Over-generalization—generic instructions may lack specificity

Automation must be paired with validation. Human oversight ensures quality.

19.9 Common Mistakes

Several common mistakes can undermine the effectiveness of the spec-to-instruction transformation:

19.9.1 1. Mixing Spec and Instruction

  • Blurring definition and execution—combining both in one layer
  • Leads to confusion—hard to separate concerns
  • Makes maintenance difficult—changes affect both definition and execution

Solution: Keep spec and instruction separate. Each has a clear role.

19.9.2 2. Skipping the Transformation

  • Using raw spec as prompt—directly feeding spec to the system
  • Results become unstable—system may interpret spec inconsistently
  • Loses benefits of instruction—no standardized execution protocol

Solution: Always transform spec into instruction before execution.

19.9.3 3. Losing Constraints

  • Not translating constraints properly—critical requirements are dropped
  • Leads to unpredictable outputs—system violates important boundaries
  • Reduces reliability—system doesn’t meet requirements

Solution: Systematically translate each constraint into enforceable rules.

19.9.4 4. Overcomplicating Instructions

  • Adding unnecessary detail—cluttering the instruction with irrelevant information
  • Reduces clarity—main intent is obscured
    • Makes maintenance difficult—hard to understand and modify

Solution: Include only what’s necessary for reliable execution.

19.10 Instruction as a Derived Artifact

Instruction should not be the starting point of system development. It should be:

Derived from spec

19.10.1 Incorrect Approach

  • Start with prompt—begin with natural language requests
  • Adjust until it works—tweak prompts through trial and error
  • No underlying spec—no formal definition of requirements

This approach is ad hoc and difficult to scale.

19.10.2 Correct Approach

  • Define spec first—create a formal specification
  • Generate instruction from spec—systematic transformation
  • Refine spec as needed—improve definition, not just prompts

This approach is systematic and scalable.

Stable systems are built from stable definitions. Instructions should always be derived from specs, not created independently.

19.11 Toward a Compilation Pipeline

We can now define a complete pipeline that spans from intent to execution:

Intent → Spec → Instruction → Execution → Evaluation

Each stage transforms the problem into a more executable form:

  1. Intent → human goal and aspiration
  2. Spec → structured, precise definition
  3. Instruction → executable protocol
  4. Execution → actual system behavior
  5. Evaluation → assessment of results

This pipeline provides a complete framework for building reliable AI systems.

19.12 Final Insight

The transition from spec to instruction is where:

  • Definition becomes action—abstract concepts become concrete operations
  • Structure becomes behavior—static definitions produce dynamic results

It is a critical step that enables systems to actually do work rather than just exist as ideas or specifications.

A system is only as good as its ability to execute its specification.

19.13 Closing Statement

Spec defines what should happen. Instruction ensures that it does.

The clearer the transformation, the more reliable the execution.

19.14 Key Insight

The spec-to-instruction transformation is analogous to compilation. It converts declarative definitions into executable protocols. The process involves extracting task, mapping input/output, translating constraints, and encoding evaluation signals.

19.15 Action Hint

Always transform specs into instructions systematically. Don’t skip this step or mix spec and instruction. Create a clear transformation process and follow it consistently. Document how you translate each element of the spec into the instruction.

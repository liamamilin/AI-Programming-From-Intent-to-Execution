2  The Shift: From Coding to Defining

2.1 Programming Has Always Evolved

Programming has never been static. Throughout the history of computing, it has continuously evolved through layers of abstraction, each transformation moving developers further from the raw machinery and closer to the problem domain itself.

This evolution is not about replacing previous skills—it is about shifting the boundary of responsibility. Each new layer of abstraction does not eliminate the need for understanding lower levels; rather, it allows developers to focus their cognitive energy on higher-level concerns while delegating lower-level details to automated systems.

2.1.1 Phase 1 — Machine-Level Programming

In the earliest days of programming, developers worked directly with hardware through assembly language and machine code. Every instruction had to be explicitly specified, and every bit of memory had to be manually managed.

  • Direct manipulation of hardware registers and memory addresses
  • Writing assembly instructions mapped one-to-one to machine operations
  • Manual management of every computational resource

The core challenge in this era was straightforward but demanding:

Control the machine precisely

Programmers needed intimate knowledge of the underlying architecture—register configurations, memory layout, instruction timing. A single mistake could crash the system or produce unpredictable results. Success required an almost mechanical precision in thinking.

2.1.2 Phase 2 — High-Level Languages

The introduction of high-level programming languages like C, Java, and Python represented a massive leap forward. These languages abstracted away many hardware details, allowing developers to express logic in terms that resembled human thought rather than machine architecture.

  • Variables and data types instead of memory addresses
  • Control structures like loops and conditionals instead of jump instructions
  • Functions and procedures instead of raw instruction sequences

The core challenge shifted from machine control to something more intellectual:

Express logic clearly

Developers no longer needed to worry about register allocation or memory alignment. Instead, they focused on:

  • Designing algorithms that efficiently solved problems
  • Structuring control flow through clear conditional logic
  • Organizing data into appropriate structures
  • Writing code that other humans could understand and maintain

This was the first major shift toward definition over implementation—the language became a medium for expressing computational logic rather than specifying machine behavior.

2.1.3 Phase 3 — Frameworks and Libraries

As software systems grew more complex, developers realized that they were repeatedly solving similar problems. This led to the rise of frameworks and libraries—pre-built components that encapsulated common patterns and functionality.

  • Web frameworks handling routing, authentication, and database connections
  • Object-relational mappers abstracting database interactions
  • SDKs providing standardized interfaces to external services

The core challenge evolved once again:

Compose existing components

Instead of building everything from scratch, developers became integrators and composers:

  • Reusing established patterns rather than reinventing solutions
  • Integrating multiple libraries and frameworks into coherent systems
  • Focusing on the unique application logic rather than infrastructure concerns
  • Making architectural decisions about which components to use and how to connect them

This phase introduced a new way of thinking: developers were no longer building systems atomically—they were assembling them from well-defined parts.

2.1.4 Phase 4 — Cloud and APIs

The cloud revolution further abstracted away infrastructure concerns. Developers could now deploy and scale applications without managing physical servers, and could access powerful capabilities through well-defined APIs.

  • Infrastructure as a Service providing on-demand computing resources
  • Platform as a Service offering complete application hosting environments
  • API ecosystems connecting systems across organizational boundaries

The core challenge shifted to a new dimension:

Orchestrate distributed systems

Developers became system coordinators rather than implementers:

  • Calling services through APIs instead of building everything internally
  • Managing flows of data across multiple distributed components
  • Designing resilient architectures that could handle partial failures
  • Thinking in terms of system-level behavior rather than program-level logic

This phase represented the culmination of decades of abstraction—developers could now build remarkably complex systems while rarely touching low-level implementation details.

2.2 A Pattern Across History

Looking across all these phases of evolution, a clear and consistent pattern emerges:

Each evolution reduces the need to focus on “how things are done” and increases the importance of defining “what should be done.”

Abstraction consistently moves upward, taking developers closer to the problem domain and further from implementation details. With each shift, the boundary of responsibility moves:

  • From low-level execution → to high-level definition
  • From specifying steps → to describing outcomes
  • From managing details → to orchestrating systems

This pattern suggests that the history of programming is the history of moving the developer’s attention up the abstraction ladder. The LLM era represents not a break from this pattern, but its acceleration and transformation.

2.3 The LLM Breakpoint

Large Language Models introduce a fundamentally different kind of abstraction—one that does not merely add another layer to the stack, but fundamentally restructures the relationship between human intent and machine execution.

Previous abstractions—assembly languages, high-level languages, frameworks, cloud services—all required developers to specify how a task should be accomplished, albeit at increasing levels of abstraction. An algorithm still needed to be designed, control flow still needed to be specified, components still needed to be assembled.

LLMs abstract something more fundamental: the act of implementation itself.

2.3.1 What Changes?

To understand this shift, consider what changes in the development workflow:

Previously: - You define the logic that solves a problem - You break that logic into implementable steps - You write code that implements each step - You debug and refine the implementation

Now: - You describe the desired behavior in natural language - You specify constraints and requirements - The system generates the implementation automatically - You evaluate and refine the definition

The key difference is that implementation becomes a derived activity rather than a primary one. The focus shifts from crafting code to crafting definitions.

2.3.2 This is not just another layer

It is crucial to understand that LLMs do not represent a continuation of the existing abstraction hierarchy. They introduce a discontinuity—a break from the pattern of gradual abstraction.

  • Frameworks abstract code structure, but you still write the logic
  • Libraries provide functionality, but you still compose them manually
  • Cloud services abstract infrastructure, but you still orchestrate systems

LLMs abstract code creation itself. When you describe what you want in sufficient detail, the system can generate the implementation without your direct involvement in the coding process.

This is a discontinuity, not a continuation. It changes not just what we do, but how we think about what programming is.

2.4 From Writing Code to Describing Systems

This shift manifests in concrete changes to how developers work. Consider a practical example: building a web scraper.

In the traditional approach: - Write HTTP request handling code - Implement HTML parsing logic - Design data extraction patterns - Handle edge cases and errors - Manage concurrent requests - Store results in a database

In the LLM-enhanced approach: - Describe the goal: “Extract product information from e-commerce pages” - Specify constraints: “Handle pagination, extract price and title, avoid duplicates” - Define output format: “JSON with product_id, title, price, url” - The system generates and executes the scraping logic

The developer’s activity has shifted from writing code to describing systems. Instead of implementing functionality, they define:

  • Intent—what they want to achieve
  • Constraints—what boundaries must be respected
  • Outcomes—what success looks like
  • Context—under what conditions the system operates

2.5 The Shift: From How to What

We can summarize this transformation with a simple yet profound change in perspective:

Before: How do we build this?
Now:    What exactly should be built?

This shift is not merely linguistic—it represents a fundamental change in the mental model of programming.

2.5.1 “How” does not disappear

It is important to clarify that this shift does not mean implementation ceases to matter. Implementation still exists, and systems still execute logic through code. The difference lies in responsibility:

  • Implementation details are increasingly handled by AI systems
  • Developers no longer need to specify every step manually
  • The “how” becomes a derived concern rather than a primary one

The system still needs to know how to accomplish tasks—but that knowledge is now encoded in the AI’s training and inference capabilities rather than being explicitly programmed by the developer.

2.5.2 “What” becomes the bottleneck

As execution becomes easier and more automated, a new difficulty emerges that was previously hidden by the complexity of implementation:

  • Vague goals that resist precise specification
  • Incomplete definitions that miss critical requirements
  • Hidden assumptions that lead to unexpected behaviors
  • Ambiguous constraints that allow multiple interpretations

The problem has shifted from technical capability to conceptual clarity. The question is no longer:

“Can we build it?”

But rather:

“Do we know exactly what we are building?”

This is a profound shift because it changes the source of difficulty. In traditional programming, challenges were primarily technical—how to make the code work, how to handle edge cases, how to optimize performance. In the LLM era, challenges become primarily conceptual—how to define what we want with sufficient precision that a system can reliably implement it.

2.6 A Change in Responsibility

This shift in focus fundamentally changes the role of the developer. The identity and daily activities of programmers evolve in important ways:

From: - Implementer—translating designs into working code - Debugger—finding and fixing errors in implementation - Code optimizer—improving efficiency and performance

To: - Problem definer—clarifying what problem should be solved - System designer—specifying how components should interact - Specification author—writing clear, executable definitions

This is not a demotion of the developer’s role—it is an elevation. Instead of being a translator between requirements and implementation, the developer becomes a designer of systems and an architect of behavior.

2.7 A New Kind of Skill

In this new paradigm, the most valuable skills are different from those traditionally prized:

  • Defining goals precisely—moving from vague ideas to specific, measurable objectives
  • Structuring problems clearly—breaking complex problems into well-defined components
  • Expressing constraints explicitly—articulating boundaries and limitations clearly
  • Designing evaluation criteria—defining how success will be measured

These are not traditional “coding skills” in the sense of syntax mastery or algorithm implementation. They are definition skills—the ability to think clearly about what should exist and to express that clarity in forms that systems can execute.

A developer who excels in this new era is someone who can: - Take a vague requirement and make it precise - Identify hidden assumptions and make them explicit - Design systems that behave predictably - Evaluate whether a system truly solves the intended problem

2.8 Why This Matters

Understanding and embracing this shift is not optional—it is essential for staying relevant in the evolving landscape of software development.

If you continue to operate under the old model: - You will focus your energy on implementation details that AI systems can handle - You will underutilize the capabilities of AI systems by using them as simple code generators - You will remain constrained by the limits of manual execution and human cognition - You will compete with AI on its strengths rather than leveraging its capabilities

If you adopt the new model: - You will shift your focus to definition, where human insight remains uniquely valuable - You will leverage AI for execution, amplifying your capabilities rather than competing with them - You will operate at a higher level of abstraction, building systems rather than writing code - You will position yourself as a system designer rather than an implementer

The choice is not between human and AI—it is between leveraging AI as a tool for definition versus using AI as a tool for implementation. The former approach yields exponentially greater leverage.

2.9 The Core Insight

The evolution of programming has always been about abstraction—moving away from implementation details and toward the problem domain. But the LLM era introduces a qualitatively new level of abstraction:

The abstraction of implementation itself.

This is not just another step in a long progression—it is a transformation of what programming means. We are moving from an era where programmers specify how to accomplish tasks to an era where they define what should exist.

This transformation requires us to reconsider not just our tools and techniques, but our fundamental understanding of what programming is and what programmers do.

2.10 Final Statement

Programming is no longer primarily about writing code. It is about defining systems—specifying intent, structuring behavior, and designing outcomes in ways that enable AI systems to execute reliably.

The future programmer is not the one who can implement everything,
but the one who can define exactly what should exist.

This shift is not a threat to programmers—it is an elevation of the role. The most valuable developers will be those who master the art of definition, who can think clearly about what should exist and express that clarity in forms that systems can execute.

2.11 Key Insight

Programming has evolved from controlling machines to defining systems. The bottleneck has shifted from execution to definition—clarity of intent is now the primary constraint.

2.12 Action Hint

For your next project, spend 80% of your time defining what you want to build before writing any code. Practice articulating goals, constraints, and success criteria in explicit, unambiguous language.

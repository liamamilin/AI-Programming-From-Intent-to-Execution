2  转变：从编码到定义（The Shift: From Coding to Defining）

2.1 编程从未停止演化

编程从来不是静止的。在计算的整个历史中，它通过抽象（abstraction）层不断演化，每一次变革都将开发者从原始机器推向问题领域本身。

这种演化不是取代以前的技能——而是移动责任的边界。每一层新的抽象都没有消除理解底层需求；相反，它允许开发者将认知能量聚焦在高层关注点上，同时将底层细节委托给自动化系统。

2.1.1 第一阶段：机器级编程（Phase 1 — Machine-Level Programming）

在编程的最早期，开发者通过汇编语言和机器码直接与硬件工作。每条指令都必须被显式指定，每比特内存都必须被手动管理。

  • 直接操作硬件寄存器和内存地址
  • 编写与机器操作一一映射的汇编指令
  • 手动管理每个计算资源

这个时代的核心挑战直接而苛刻：

精确控制机器

程序员需要深入了解底层架构——寄存器配置、内存布局、指令时序。一个错误可能使系统崩溃或产生不可预测的结果。

2.1.2 第二阶段：高级语言（Phase 2 — High-Level Languages）

C、Java、Python 等高级编程语言的引入代表了巨大的飞跃。这些语言抽象了许多硬件细节，允许开发者用更接近人类思维而非机器架构的方式表达逻辑。

  • 变量和数据类型代替内存地址
  • 循环和条件等控制结构代替跳转指令
  • 函数和过程代替原始指令序列

核心挑战从机器控制转向了更智力性的东西：

清晰地表达逻辑

开发者不再需要担心寄存器分配或内存对齐。他们专注于设计高效解决问题的算法、组织清晰的控制流、将数据组织到适当的结构中、编写其他人能够理解和维护的代码。

2.1.3 第三阶段：框架和库（Phase 3 — Frameworks and Libraries）

随着软件系统变得更加复杂，开发者意识到他们在重复解决类似的问题。这导致了框架和库的兴起——封装了常见模式和功能的预制组件。

  • Web 框架处理路由、认证和数据库连接
  • ORM 抽象数据库交互
  • SDK 提供外部服务的标准化接口

核心挑战再次演化：

组合现有组件

开发者不再从头构建一切，而是成为集成者和组合者：复用既有模式、集成多个库和框架、关注独特的应用逻辑而非基础设施问题、做出关于使用哪些组件以及如何连接它们的架构决策。

2.1.4 第四阶段：云和 API（Phase 4 — Cloud and APIs）

云革命进一步抽象了基础设施问题。开发者无需管理物理服务器即可部署和扩展应用，并通过定义良好的 API 访问强大的能力。

  • 基础设施即服务提供按需计算资源
  • 平台即服务提供完整的应用托管环境
  • API 生态系统连接跨组织边界的系统

核心挑战转向了一个新维度：

编排分布式系统

开发者成为系统协调者而非实现者：通过 API 调用服务而非内部构建一切、管理跨多个分布式组件的数据流、设计能够处理部分失败的韧性架构、以系统级行为而非程序级逻辑的方式思考。

2.1.5 贯穿历史的模式（A Pattern Across History）

回顾所有演化阶段，一个清晰一致的模式浮现出来：

每次演化都减少了对”事情如何完成”的关注，增加了对”应该做什么”的定义的重要性。

抽象不断向上移动，将开发者带到离问题领域更近、离实现细节更远的地方。每次转变，责任的边界都在移动：

  • 从低层执行 → 到高层定义
  • 从指定步骤 → 到描述结果
  • 从管理细节 → 到编排系统

2.1.6 LLM 断点（The LLM Breakpoint）

大语言模型引入了一种根本不同的抽象——不只向栈中添加另一层，而是从根本上重构了人类意图与机器执行之间的关系。

先前的抽象——汇编语言、高级语言、框架、云服务——都要求开发者指定任务如何完成，尽管抽象层级在不断提高。算法仍然需要设计，控制流仍然需要指定，组件仍然需要组装。

LLM 抽象了更基础的东西：实现本身的行为。

2.1.7 什么变了？

要理解这一转变，考虑开发工作流中发生的变化：

以前： 你定义解决问题的逻辑 → 将逻辑分解为可实现的步骤 → 编写实现每个步骤的代码 → 调试和优化实现

现在： 你用自然语言描述期望的行为 → 指定约束和要求 → 系统自动生成实现 → 你评估和优化定义

关键区别是：实现成为派生活动而非主要活动。焦点从编写代码转向编写定义。

2.1.8 这不是另一层抽象

LLM 不代表对现有抽象层次的延续。它们引入了一个不连续性——一个与渐进式抽象模式的决裂。

  • 框架抽象代码结构，但你仍然写逻辑
  • 库提供功能，但你仍然手动组合
  • 云服务抽象基础设施，但你仍然编排系统

LLM 抽象了代码创建本身。 当你用足够细节描述想要什么时，系统可以在不直接参与编码过程的情况下生成实现。

这是一个不连续性，而非延续性。它改变的不仅是我们做什么，还有我们对编程本质的思考方式。

2.1.9 从写代码到描述系统（From Writing Code to Describing Systems）

用一个实际例子来说明：构建一个 Web 爬虫。

传统方式： 编写 HTTP 请求处理代码 → 实现 HTML 解析逻辑 → 设计数据提取模式 → 处理边界情况和错误 → 管理并发请求 → 将结果存储在数据库中

LLM 增强方式： 描述目标：“从电商页面提取产品信息” → 指定约束：“处理分页、提取价格和标题、避免重复” → 定义输出格式：“JSON，包含 product_id、title、price、url” → 系统生成并执行爬取逻辑

开发者的活动已从编写代码转移到描述系统。他们定义：意图、约束、结果、上下文。

2.1.10 核心洞见（The Core Insight）

编程的演化一直是关于抽象——远离实现细节，走向问题域。但 LLM 时代引入了一种质的新抽象层次：

对实现本身的抽象。

这不是漫长进程中的又一步——它是对编程含义的转变。我们正在从一个程序员指定如何完成任务的时代，走向一个他们定义什么应该存在的时代。

未来的程序员不是能实现一切的人，而是能精确定义应该存在什么的人。

这种转变不是对程序员的威胁——而是对角色的提升。最有价值的开发者将是那些掌握定义艺术的人，那些能清晰思考应该存在什么并以系统可执行的形式表达清晰性的人。

26  Debugging in the LLM Era

26.1 The Shift in Debugging

In traditional programming, debugging has a clear, well-defined meaning:

  • Finding bugs in code—locating errors in implementation
  • Fixing logic errors—correcting flawed reasoning or algorithms
  • Tracing execution paths—following program flow to identify issues

The object of debugging is clear and specific:

Code

In the LLM era, this changes fundamentally. Systems are no longer defined primarily by code. They are defined by:

  • Spec—the specification of what the system should do
  • Instruction—the protocol for how the system should execute
  • Context—the information available to the system at execution time

Debugging is no longer about fixing code. It is about fixing definitions.

26.2 What Are We Debugging Now?

When an LLM system produces incorrect output, the problem is rarely:

  • A syntax error—grammatical or structural mistake in code
  • A runtime exception—program crashes or errors during execution
  • A logical bug—error in algorithmic reasoning

Instead, the problem is usually:

  • Ambiguity—unclear or multiple interpretations of requirements
  • Missing constraints—insufficient boundaries on acceptable behavior
  • Poor structure—lack of clear organization or framework

26.2.1 New Debug Targets

In the LLM era, debugging focuses on:

  • Spec—unclear or incomplete definition of requirements
  • Instruction—poorly designed execution protocol
  • Context—irrelevant or missing information

The bug is often in what we said, not how it was executed. The system follows our definition—when the definition is flawed, the result is flawed.

26.3 Types of Failures

LLM systems can fail in different ways, each pointing to different underlying issues:

26.3.1 1. Incorrect Output

  • Wrong answer—response doesn’t match expected result
  • Irrelevant content—output addresses different question
  • Hallucination—system invents information

Likely causes: - Unclear spec—requirements not well-defined - Weak instruction—execution protocol insufficient - Poor context—missing or irrelevant information

26.3.2 2. Inconsistent Behavior

  • Same input → different outputs—results vary unpredictably
  • Random variation—no clear pattern in differences
  • Context-dependent inconsistency—behavior changes with session

Likely causes: - Weak constraints—insufficient boundaries - Poor instruction design—lack of structure - Context variability—different context construction

26.3.3 3. Format Violations

  • Output does not match expected structure—wrong JSON, missing fields
  • Incorrect format—wrong data types, organization
  • Parsing errors—output can’t be processed downstream

Likely causes: - Poor output design—unclear format requirements - Weak instruction—inadequate format specification - Missing validation—no format checking

26.3.4 4. Missing Information

  • Incomplete responses—output lacks required elements
  • Partial answers—only addresses part of query
  • Omitted details—important information missing

Likely causes: - Incomplete spec—requirements not fully specified - Weak constraints—no requirement for completeness - Poor context—missing source information

Each type of failure points to a different layer—spec, instruction, or context—where the fix should be applied.

26.4 Debugging as Diagnosis

Debugging begins with diagnosis—identifying where the problem originates.

26.4.1 Key Question

Where is the problem?

26.4.2 Layered Diagnosis

Diagnosing LLM systems requires examining each layer:

26.4.2.1 Spec Layer

  • Is the goal clearly defined?
  • Are constraints complete?
  • Are success criteria explicit?
  • Is there ambiguity in requirements?

Questions to ask: - What exactly should the system do? - What are the boundaries of acceptable behavior? - How do we know when the system is correct?

26.4.2.2 Instruction Layer

  • Is the task explicit?
  • Is the output structured?
  • Are constraints enforced?
  • Is the protocol clear?

Questions to ask: - How does the instruction define the task? - What format is expected for output? - What rules constrain behavior?

26.4.2.3 Context Layer

  • Is relevant information included?
  • Is noise removed?
  • Is context consistent?
  • Is retrieval effective?

Questions to ask: - What information is available to the system? - Is the right information being retrieved? - Is context constructed consistently?

Correct diagnosis determines correct fixes. Debugging is systematic, not random.

26.5 Debugging Spec

When the issue is in the spec, the problem is at the definition level—the system is trying to do the wrong thing.

26.5.1 Symptoms

  • Outputs are generally off-target—system consistently misunderstands requirements
  • System misunderstands intent—results don’t match original goals
  • Behavior is systematically wrong—same error pattern across multiple runs

26.5.2 Fixes

  • Clarify objective—make goals more explicit and precise
  • Add constraints—define boundaries and limitations
  • Define success criteria—specify how to evaluate correctness
  • Remove ambiguity—eliminate multiple interpretations

26.5.3 Example

Spec:

Summarize the document

Problem: Too vague—what kind of summary? How long? What to include?

Improved Spec:

Summarize the document into 5 key points, each under 20 words, focusing on main arguments and excluding minor details

Result: System produces summaries that match requirements more precisely.

Improve definition, not execution. When the spec is wrong, fixing execution won’t help.

26.6 Debugging Instruction

When the issue is in the instruction, the problem is at the protocol level—the system doesn’t know how to do what’s asked.

26.6.1 Symptoms

  • Output format is inconsistent—results vary in structure
  • Behavior varies widely—same instruction produces different results
  • Format violations—output doesn’t match expected structure

26.6.2 Fixes

  • Define output structure—specify exact format requirements
  • Add explicit rules—create clear constraints on behavior
  • Simplify task—break complex instructions into simpler steps
  • Standardize format—use consistent data structures

26.6.3 Example

Instruction:

Explain the result

Problem: Too vague—what kind of explanation? How long? What format?

Improved Instruction:

Provide a 3-step explanation in bullet points, each under 15 words, covering: (1) what was found, (2) why it matters, (3) what to do next

Result: System produces consistent, structured explanations.

26.7 Debugging Context

When the issue is in context, the problem is at the information level—the system doesn’t have the right information.

26.7.1 Symptoms

  • Irrelevant answers—response doesn’t address question
  • Missing key information—important facts omitted
  • Context-dependent failures—works sometimes, fails others

26.7.2 Fixes

  • Add relevant context—include information system needs
  • Remove noise—eliminate irrelevant information
  • Improve retrieval—enhance search and selection
  • Standardize context—consistent context construction

Context determines what the system sees. Better context leads to better results.

26.8 Debugging as Iteration

Debugging is not a one-time fix. It follows a loop of continuous improvement:

Observe → Diagnose → Modify → Test → Repeat

Each iteration:

  • Reduces ambiguity—definition becomes clearer
  • Improves stability—behavior becomes more consistent
  • Moves toward correctness—results better meet requirements

26.8.1 Iterative Process

  1. Observe—identify the failure or issue
  2. Diagnose—determine which layer has the problem
  3. Modify—adjust spec, instruction, or context
  4. Test—run system to verify fix
  5. Repeat—continue until issue resolved

26.9 Avoiding Common Pitfalls

Several common mistakes undermine effective debugging:

26.9.1 1. Blaming the Model

  • Assuming the system is flawed—blaming the LLM for poor results
  • Ignoring definition issues—overlooking problems in spec or instruction
  • Changing parameters without diagnosis—tuning settings without understanding cause

Solution: Focus on definition first. The model follows your definition—if definition is wrong, output will be wrong.

26.9.2 2. Random Adjustments

  • Changing parameters without diagnosis—tweaking settings randomly
  • Trial and error—making changes without understanding impact
  • No systematic approach—jumping between different fixes

Solution: Diagnose first, then fix. Identify the root cause before making changes.

26.9.3 3. Overfitting Fixes

  • Fixing one case but breaking others—optimizing for specific inputs
  • Narrow focus—addressing symptoms not causes
  • Fragile solutions—fixes that break easily

Solution: Understand the underlying problem. Fix at the appropriate layer.

26.9.4 4. Mixing Layers

  • Changing instruction when spec is the problem—treating symptoms not causes
  • Adjusting context when instruction is wrong—wrong layer for fix
  • Modifying everything at once—no way to isolate what works

Solution: Identify which layer has the problem. Fix at that layer only.

Debugging requires discipline, not guesswork. Systematic diagnosis leads to effective fixes.

26.10 Debugging Tools and Techniques

Effective debugging requires appropriate tools and methods:

26.10.1 1. Test Cases

  • Define inputs and expected outputs—create test scenarios
  • Cover edge cases—test unusual or boundary conditions
  • Document behavior—track what works and what doesn’t

Purpose: Understand system behavior systematically

26.10.2 2. Controlled Inputs

  • Reduce variability—use consistent, controlled inputs
  • Isolate issues—test one variable at a time
  • Reproduce failures—create reliable test cases

Purpose: Make debugging predictable and repeatable

26.10.3 3. Logging

  • Track inputs, outputs, context—record what happens at each stage
  • Monitor performance—track latency, cost, success rates
  • Identify patterns—find recurring issues

Purpose: Understand system behavior and identify problems

26.10.4 4. Evaluation Metrics

  • Measure performance—quantify system behavior
  • Track improvement—monitor progress over time
  • Set benchmarks—establish standards for success

Purpose: Make debugging objective and measurable

Measurement enables debugging. Without metrics, debugging is guesswork.

26.11 Debugging as Understanding

Debugging improves more than just the system—it improves understanding:

Each failure reveals:

  • Missing constraints—boundaries that weren’t specified
  • Hidden assumptions—implicit requirements that weren’t stated
  • Weak definitions—specifications that needed clarification

Debugging is learning:

  • The system—how it behaves and what it needs
  • The problem—what the real requirements are
  • Your understanding—how to define problems more clearly

Every bug is a signal. It tells you something about the definition that needs to be clarified.

26.12 From Debugging Code to Debugging Meaning

The fundamental shift in debugging:

26.12.1 Before

  • Debug logic—find errors in reasoning or implementation
  • Trace execution—follow program flow to identify issues
  • Fix code—correct syntax or logic errors

Focus: How the system is implemented

26.12.2 Now

  • Debug meaning—clarify what the system should do
  • Refine definition—improve spec and instruction
  • Adjust context—optimize information available to system

Focus: What the system should do

This requires:

  • Precision in language—clear, unambiguous definitions
  • Clarity in definition—explicit requirements and constraints
  • Awareness of system behavior—understanding how systems interpret definitions

26.13 Final Insight

Debugging in the LLM era is not about fixing what the system did.

It is about clarifying what the system should have done.

The system reflects your definition. If the output is wrong, the definition is incomplete. Debugging is the process of completing and refining that definition.

26.14 Closing Statement

As programming shifts from implementation to definition, debugging shifts with it.

You are no longer debugging code. You are debugging your understanding.

26.15 Key Insight

Debugging in the LLM era focuses on fixing definitions—spec, instruction, and context—rather than code. The process involves systematic diagnosis to identify which layer has the problem, then applying targeted fixes at that layer.

26.16 Action Hint

Diagnose first, then fix. Identify whether the problem is in spec, instruction, or context. Use test cases, controlled inputs, logging, and evaluation metrics to understand system behavior. Avoid blaming the model—focus on definition. Debug iteratively: observe, diagnose, modify, test, repeat.

30  构建 AI 工具（Building an AI Tool）

30.1 工具作为产品

AI 工具（tool）封装特定能力，以便其他系统可以调用。工具是 AI 系统构建中的构建块——独立的、可复用的功能单元。

与传统软件工具不同，AI 工具通常涉及概率性处理。它们的输出不是确定性的，而是基于模型推理的结果。这给工具设计带来了独特的挑战：你需要定义清晰的接口（interface），同时接受内部行为的不确定性。

本章使用最小循环（Minimal Loop）方法构建一个 AI 驱动的代码评论工具（code review tool）。

30.2 定义工具意图（Define Tool Intent）

意图： 构建一个代码评论工具——给定代码片段，提供关于风格、潜在 bug 和优化建议的反馈。

精化后的目标： - 目标：在代码审查过程中提供即时的、有针对性的反馈 - 对象：Python 和 JavaScript 代码片段（其他语言逐步增加） - 上下文：开发者提交代码时的即时反馈场景 - 约束：不执行代码（仅静态分析），每条评论必须引用具体行号，不提供虚假的安全警报 - 成功标准：建议被开发者采纳率 > 60%，误报率 < 10%

30.3 工具 Spec

输入： 代码字符串（最多 2000 字符），编程语言，可选的编码风格指南（style guide）。

输出： 结构化评论列表，每条包含：行号、严重性、消息、改进建议。

[
  {
    "line": 15,
    "severity": "warning",
    "message": "使用 == 而非 === 可能造成类型转换问题",
    "suggestion": "用 === 替代 =="
  }
]

严重性等级（Severity Levels）： - error：确定会出错的代码（安全漏洞、语法错误） - warning：可能出问题的代码（类型转换、未定义变量） - suggestion：改进建议（性能优化、代码风格）

约束： 不执行代码，仅基于静态分析（static analysis），尊重语言特定的最佳实践，保持每条评论简洁。

边界情况： 空代码（返回空列表），无法识别语言（使用通用规则），代码包含注释（跳过注释）。

30.4 工具架构（Tool Architecture）

主流程：

代码输入 → 语言检测 → 解析结构 → 分析模式 → 生成评论 → 格式化输出

模块： - 语言检测器（Language Detector）：识别输入代码的编程语言。基于文件扩展名、代码特征和魔数（shebang 等）。 - 结构分析器（Structure Analyzer）：解析代码结构，提取函数、类、变量声明。构建抽象语法树（AST）概览。 - 模式分析器（Pattern Analyzer）：对照最佳实践检查代码。识别常见反模式（antipatterns）和潜在漏洞。 - 评论生成器（Comment Generator）：将分析结果转化为结构化评论。添加行号引用和严重性评级。 - 格式化器（Formatter）：将评论格式化为统一输出。支持 JSON 和格式化文本输出。

30.5 指令示例（Instruction Example）

角色：你是一个代码评审专家。你有 15 年的软件开发经验，精通多种编程语言的最佳实践。

任务：审查提供的代码片段，识别潜在问题、反模式和改进机会。

输入：代码字符串 + 编程语言名称

输出要求：
以 JSON 数组格式输出每条评论，包含：
- "line": 行号（整数，必须精确）
- "severity": "error" | "warning" | "suggestion"
- "message": 简洁的问题描述（< 100 字符）
- "suggestion": 改进后的代码示例（可选）

约束：
- 只进行静态分析，不执行代码
- 优先报告错误和潜在安全问题
- 每条评论要有具体的行号引用
- 不报告虚假问题（宁可漏报不可误报）

示例：
输入：var x = "hello";
输出：[{"line":1, "severity":"suggestion", "message":"使用 const 替代 var", "suggestion":"const x = 'hello';"}]

30.6 评估与精化（Evaluate and Refine）

测试数据集： 20 个代码样本，包含已知问题（错误、警告、建议各 5+ 个），5 个干净代码（测试误报率）。

评估指标： - 问题检出率 > 80%（已知问题的检测比例） - 误报率 < 10%（标记为问题但不正确的比例） - 采纳率 > 60%（开发者在实际使用中采纳建议的比例）

迭代策略： 基于误报分析精化指令，添加更多特定模式的边界条件，扩展支持语言列表。

30.7 集成与部署

打包： 将工具封装为 API 端点、CLI 命令或 IDE 插件。

  • API：REST 端点，接受 POST 请求返回 JSON 评论
  • CLI：review-tool file.py 或 review-tool --stdin
  • IDE 集成：通过语言服务器协议（LSP）集成到编辑器

文档： 清晰的输入/输出规范，可直接运行的示例代码，错误码和解释。

版本管理： 语义化版本，变更日志，发布说明。

30.8 小结

AI 工具是 AI 系统的基本构建块。通过将能力封装为具有清晰 Spec 的工具，你可以构建可复用、可测试、可组合的功能单元。

构建良好 AI 工具的关键是：清晰的接口定义、精确的指令、系统化的评估和迭代。掌握了工具构建，你就为构建更复杂的 AI 系统奠定了坚实的基础。

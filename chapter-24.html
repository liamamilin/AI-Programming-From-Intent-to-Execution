25  Reliability & Evaluation

25.1 From Capability to Reliability

A system that works once is not enough. A prototype that produces a correct output in controlled conditions is not a production system.

Real systems must be:

  • Consistent—producing similar results for similar inputs
  • Predictable—behaving as expected across many executions
  • Measurable—performance can be quantified and tracked

This introduces a new requirement:

Reliability — the ability to produce correct results consistently

25.2 What is Reliability?

Reliability is the property that:

Given similar inputs, the system produces outputs that consistently meet defined success criteria

Reliability is not about being perfect or never making mistakes. It is about:

  • Consistency—similar inputs produce similar outputs
  • Correctness—outputs meet requirements and expectations
  • Stability—behavior does not degrade over time

25.2.1 Key Dimensions

  • Consistency—similar inputs → similar outputs
  • Correctness—outputs meet requirements
  • Stability—behavior does not degrade over time

Reliability is not about perfection. It is about consistency within defined bounds.

25.3 Why Reliability is Hard in LLM Systems

LLM-based systems face unique reliability challenges that make achieving consistency difficult:

LLMs are inherently:

  • Probabilistic—outputs are sampled from a distribution, not deterministic
  • Context-dependent—results vary based on input context and history
  • Sensitive to input variation—small changes can produce large differences

This leads to:

  • Output variability—same prompt can produce different answers
  • Edge case failures—unusual inputs break the system
  • Hidden instability—problems appear only under specific conditions

Reliability must be engineered. It does not emerge automatically in probabilistic systems.

25.4 Sources of Instability

Understanding why systems become unreliable helps design better solutions:

25.4.1 1. Weak Spec

  • Ambiguous definitions—requirements are unclear or incomplete
  • Missing constraints—no boundaries on acceptable behavior
  • Weak success criteria—no clear measure of correctness

Result: System interprets requirements inconsistently

25.4.2 2. Poor Instruction Design

  • Inconsistent formats—output structure varies
  • Unclear task definition—system doesn’t understand what to do
  • Missing structure—no framework for execution

Result: Behavior varies between executions

25.4.3 3. Context Variability

  • Different context → different results—context affects interpretation
  • Context window limits—can’t include all relevant information
  • Context contamination—irrelevant information included

Result: Inconsistent behavior across sessions

25.4.4 4. Tool Errors

  • External API failures—services unavailable or malfunctioning
  • Incorrect data—tools return wrong information
  • Tool misuse—wrong tool used or used incorrectly

Result: Unreliable external operations

25.4.5 5. Lack of Evaluation

  • No feedback loop—can’t detect or correct errors
  • Unknown quality—don’t know if results are correct
  • Uncaught failures—errors propagate silently

Result: Problems accumulate and compound

Most instability originates upstream—in definition and design, not execution.

25.5 Principles for Stability

Achieving reliability requires applying specific principles throughout system design:

25.5.1 1. Strengthen Spec

  • Define clear constraints—explicit boundaries on behavior
  • Reduce ambiguity—eliminate multiple interpretations
  • Specify edge cases—handle unusual inputs explicitly

Result: System has clear, unambiguous requirements

25.5.2 2. Enforce Structure

  • Use structured outputs—exact format requirements
  • Standardize formats—consistent data structures
  • Validate structure—check format at every stage

Result: Predictable, parseable outputs

25.5.3 3. Control Context

  • Manage inputs carefully—validate and clean input data
  • Reduce noise—remove irrelevant information
  • Standardize context—consistent context construction

Result: Consistent processing environment

25.5.4 4. Add Redundancy

  • Multiple checks—verify results at multiple stages
  • Fallback mechanisms—alternative approaches when primary fails
  • Redundant processing—compare multiple outputs

Result: Errors caught before they affect users

25.5.5 5. Introduce Evaluation

  • Continuous validation—check every output
  • Feedback loops—use results to improve system
  • Quality metrics—track performance over time

Result: System learns and improves

Stability is achieved through layered control. Multiple mechanisms work together to ensure reliability.

25.6 What is Evaluation in This Context?

Evaluation is the process of:

Measuring whether system outputs meet defined success criteria

It provides:

  • A signal of correctness—knowing whether results are acceptable
  • A basis for iteration—information for improving the system
  • A mechanism for control—triggering corrections or retries

25.6.1 Why Evaluation Matters

Without evaluation:

  • No notion of correctness—can’t distinguish good from bad results
  • No direction for improvement—don’t know what to fix
  • No system stability—behavior can drift without detection

25.7 Automatic Evaluation

Manual evaluation does not scale for production systems. Therefore, we introduce:

Automatic Evaluation

25.7.1 Definition

Automatic evaluation is:

The use of rules, models, or metrics to assess outputs without human intervention

25.8 Types of Automatic Evaluation

Different approaches suit different evaluation needs:

25.8.1 1. Rule-Based Evaluation

  • Deterministic checks—yes/no questions about output
  • Examples:
    • JSON validity—output is valid JSON
    • Length constraints—output under specified length
    • Required fields—all mandatory fields present
  • Advantages:
    • Fast and precise
    • No false positives
    • Easy to implement
  • Disadvantages:
    • Limited scope
    • Can’t assess quality
    • Rigid

25.8.2 2. Model-Based Evaluation

  • Use another model to evaluate—judge output quality
  • Examples:
    • Relevance scoring—how relevant is output to input?
    • Hallucination detection—is output grounded in facts?
    • Sentiment analysis—is tone appropriate?
  • Advantages:
    • Can assess complex criteria
    • Flexible and adaptable
    • Can evaluate nuance
  • Disadvantages:
    • Can be expensive
    • May have biases
    • Slower than rules

25.8.3 3. Metric-Based Evaluation

  • Quantitative measures—numerical scores for quality
  • Examples:
    • Accuracy—percentage of correct outputs
    • Precision/recall—performance on classification tasks
    • Latency—response time
  • Advantages:
    • Objective and comparable
    • Trackable over time
    • Industry standards exist
  • Disadvantages:
    • May not capture all aspects of quality
    • Can be gamed
    • Need ground truth

25.8.4 4. Hybrid Evaluation

  • Combine multiple methods—use rules, models, and metrics together
  • Example:
    • Rule: Output must be JSON
    • Model: JSON must contain relevant information
    • Metric: Response time under 2 seconds
  • Advantages:
    • Comprehensive
    • Leverages strengths of each approach
    • More robust
  • Disadvantages:
    • More complex to implement
    • Higher cost
    • Requires coordination

Strong systems use layered evaluation—multiple methods working together to ensure quality.

25.9 Evaluation Pipeline

A robust system includes an evaluation pipeline that checks quality at every stage:

Output → Validation → Scoring → Decision

25.9.1 Steps

25.9.1.1 1. Validation

  • Check format—ensure output meets structural requirements
  • Ensure constraints—verify output respects defined boundaries
  • Validate completeness—check that all required elements are present

25.9.1.2 2. Scoring

  • Assess quality—assign score based on evaluation criteria
  • Measure performance—calculate metrics like accuracy or relevance
    • Rate results—determine how well output meets requirements

25.9.1.3 3. Decision

  • Accept—output meets requirements, proceed to user
  • Reject—output fails validation, trigger retry or correction
  • Flag—output is marginal, requires human review

Evaluation is not passive. It drives system behavior and ensures quality.

25.10 Feedback Loops

Evaluation enables feedback—using results to improve the system:

25.10.1 Immediate Feedback

  • Reject invalid outputs—don’t present to user
  • Trigger retries—attempt to produce better result
  • Apply corrections—automatically fix simple errors

Purpose: Ensure user receives only high-quality results

25.10.2 Long-Term Feedback

  • Improve spec—refine definitions based on evaluation results
  • Refine instructions—adjust protocols based on what works
  • Optimize parameters—tune system settings for better performance

Purpose: Continuous improvement over time

Feedback transforms systems from static to adaptive. Systems that learn from evaluation become more reliable.

25.11 Techniques for Improving Reliability

Several techniques can enhance system reliability:

25.11.1 1. Multiple Runs (Sampling)

  • Generate multiple outputs—run instruction several times
  • Select best—choose highest quality output
  • Average or aggregate—combine multiple outputs

Purpose: Increase chance of good result

25.11.2 2. Self-Consistency

  • Compare multiple answers—run same instruction multiple times
  • Choose consistent result—select output that appears most frequently
  • Identify outliers—flag unusual outputs for review

Purpose: Reduce randomness and improve stability

25.11.3 3. Verification Steps

  • Add validation stages—check output at multiple points
  • Verify correctness—confirm output meets requirements
    • Cross-check—compare against expected patterns

Purpose: Catch errors before they propagate

25.11.4 4. Guardrails

  • Enforce constraints—prevent outputs that violate rules
  • Filter inappropriate content—block harmful or incorrect outputs
  • Limit behavior—keep system within defined boundaries

Purpose: Prevent unacceptable outputs

Reliability is built through layers of verification. Multiple checks ensure quality.

25.12 Trade-offs

System design involves balancing competing requirements:

25.12.1 Reliability vs Cost

  • More checks → higher cost—additional evaluation increases computational expense
  • Fewer checks → lower cost—reduced verification saves resources
  • Balance: Choose evaluation depth based on cost constraints

25.12.2 Reliability vs Speed

  • More evaluation → slower execution—verification takes time
  • Less evaluation → faster response—reduced checks improve speed
  • Balance: Optimize for required latency while maintaining acceptable reliability

25.12.3 Reliability vs Flexibility

  • More constraints → less flexibility—strict rules limit options
  • Fewer constraints → more flexibility—loose rules allow more variation
  • Balance: Define constraints that ensure reliability without limiting necessary flexibility

System design requires balancing these factors based on specific requirements and constraints.

25.13 Common Mistakes

Several common mistakes undermine reliability:

25.13.1 1. Relying on Single Output

  • No validation—accept first output without checking
  • High risk—errors propagate to users
  • Unpredictable quality—no guarantee of correctness

Solution: Always validate outputs. Implement evaluation pipelines.

25.13.2 2. No Clear Metrics

  • Cannot measure improvement—don’t know if system is getting better
  • No baseline—can’t assess performance
  • Blind optimization—changing things without knowing if they help

Solution: Define clear metrics. Track performance over time.

25.13.3 3. Over-Reliance on Human Evaluation

  • Does not scale—can’t evaluate every output manually
  • Inconsistent—human judgment varies
  • Slow—manual evaluation is time-consuming

Solution: Implement automatic evaluation. Use human evaluation for special cases.

25.13.4 4. Ignoring Edge Cases

  • System fails under variation—unusual inputs break the system
  • Hidden failures—problems only appear in rare cases
  • False confidence—system works on test data but fails in production

Solution: Test diverse inputs. Identify and handle edge cases explicitly.

25.14 Reliability in the Full System

Reliability integrates into the full Minimal Loop pipeline:

Intent → Spec → Instruction → Execution → Evaluation → Feedback

It ensures that:

  • Each stage is validated—quality checked at every point
  • Errors are detected—problems identified early
  • Systems improve over time—feedback drives refinement

25.15 Final Insight

Reliability is not a feature added at the end. It is a property of the entire system.

A system is only as reliable as its weakest definition. Every layer—from intent to evaluation—must be designed for reliability.

25.16 Closing Statement

In the LLM era, building systems is not just about making them work. It is about making them work consistently.

Reliability is what turns capability into trust.

25.17 Key Insight

Reliability enables systems to produce correct results consistently. Key principles include strengthening spec, enforcing structure, controlling context, adding redundancy, and introducing evaluation. Automatic evaluation methods include rule-based, model-based, metric-based, and hybrid approaches.

25.18 Action Hint

Implement evaluation at every stage. Use rule-based checks for format and constraints. Use model-based evaluation for quality assessment. Track metrics over time. Build feedback loops for continuous improvement. Balance reliability with cost, speed, and flexibility. Test diverse inputs to catch edge cases.

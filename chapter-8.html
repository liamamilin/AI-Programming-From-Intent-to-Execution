9  Evaluation & Iteration

9.1 Why Execution Is Not the End

In traditional software development, execution often marks completion. When code runs successfully and produces output, the task is often considered done:

  • The code compiles without errors
  • The system produces output
  • The task is considered complete

This mental model made sense when implementation was the primary challenge. Getting code to run correctly was difficult enough that once it worked, we tended to move on.

In the LLM paradigm, this assumption no longer holds. Execution produces a result, not a guarantee of correctness or quality.

Because:

  • Outputs are probabilistic—the same input can produce different outputs
  • Behavior depends on input variation—slightly different inputs can lead to very different results
  • Systems interpret rather than strictly follow instructions—LLMs use semantic understanding, not literal rule-following

Therefore:

Execution is not the end of the process. It is the beginning of validation.

9.2 What is Evaluation?

Evaluation is the process of determining whether the system’s output satisfies the original intent and specification. It is the mechanism that closes the loop and enables systematic improvement.

Evaluation answers critical questions:

  • Is the result correct?—does it achieve the intended goal?
  • Does it meet the constraints?—does it respect all specified boundaries?
  • Does it achieve the intended outcome?—does it solve the problem we set out to solve?

Evaluation connects output back to definition, creating the feedback loop necessary for systematic improvement.

9.3 Why Evaluation Is Necessary

Without evaluation, system development is fundamentally flawed:

  • There is no notion of correctness—any output is acceptable
  • There is no direction for improvement—changes are random rather than purposeful
  • There is no system stability—behavior drifts without correction

9.3.1 In the LLM Era

The need for evaluation is even greater because execution is so easy:

  • Systems produce outputs quickly and cheaply
  • Iterations happen rapidly and frequently
  • It’s tempting to move quickly without measuring results

But without evaluation:

  • Iteration becomes random—you change things without knowing whether they help
  • Improvements are accidental—you might get better results, but you don’t know why
  • Quality cannot be controlled—you have no way to ensure consistency or reliability

Speed without evaluation leads to chaos, not progress.

9.4 Types of Evaluation

Evaluation can take multiple forms depending on the system, the context, and the requirements. Different types of evaluation serve different purposes.

9.4.1 1. Rule-Based Evaluation

Rule-based evaluation uses explicit, deterministic rules to assess outputs. It is simple, fast, and unambiguous.

Examples: - Output must be valid JSON - Response time must be under 2 seconds - Output must contain all required fields

Characteristics: - Deterministic—the same evaluation always produces the same result - Fast—can be executed automatically at scale - Limited—cannot assess qualitative aspects like relevance or quality

9.4.2 2. Human Evaluation

Human evaluation relies on subjective judgment to assess outputs. It can capture nuanced aspects that automated methods cannot.

Examples: - Is the answer helpful? - Does it align with user expectations? - Is the tone appropriate?

Characteristics: - Subjective—different humans may evaluate differently - Context-aware—can understand subtle aspects of meaning and appropriateness - Expensive—requires human time and attention - Non-scalable—difficult to apply to large volumes of data

9.4.3 3. Model-Based Evaluation

Model-based evaluation uses another AI model to assess the output. This can provide automated evaluation of qualitative aspects.

Examples: - Score relevance on a scale of 1-10 - Detect hallucinations or factual errors - Assess writing quality or clarity

Characteristics: - Automated—can be applied at scale - Nuanced—can assess qualitative aspects - Imperfect—the evaluating model may have its own biases and limitations - Requires validation—the evaluating model itself must be validated

9.4.4 4. Hybrid Evaluation

Hybrid evaluation combines multiple evaluation methods to get a more complete picture of output quality.

Examples: - Rule-based checks for structure + human review for quality - Model-based scoring for relevance + rule-based checks for format - Multiple evaluators (human and automated) with disagreement resolution

Characteristics: - Comprehensive—captures multiple dimensions of quality - Complex—requires integrating different types of feedback - More reliable—less likely to miss important issues

Strong systems rely on layered evaluation—combining multiple approaches to ensure comprehensive assessment.

9.5 Evaluation Defines Quality

Quality is not an inherent property of output. It is defined by evaluation. Without evaluation criteria:

  • “Good” becomes subjective and variable
  • Systems drift over time without correction
  • Results cannot be compared or measured

9.5.1 Example

Consider an intent to “summarize a document.”

Without evaluation: - Any summary is acceptable - No way to determine if one summary is better than another - System may produce summaries that miss important information

With evaluation: - Must include key points from the original document - Must be concise (under 100 words) - Must avoid redundancy - Must preserve the original meaning

Now quality is defined and measurable. The system can be improved systematically by refining its performance against these criteria.

Evaluation makes quality measurable and therefore improvable.

9.6 The Role of Iteration

Evaluation alone is not enough. It must lead to iteration—systematic improvement based on evaluation feedback.

9.6.1 Iteration Loop

Define → Execute → Evaluate → Refine

This loop is the engine of progress. Each cycle should move the system closer to the desired behavior.

9.6.2 Refinement typically occurs in:

  • Intent—clarifying or adjusting the goal if the original intent was wrong
  • Spec—adding missing constraints, clarifying ambiguous definitions
  • Instruction—improving the execution protocol for better results

The key insight is that most improvements happen upstream—by improving definitions, not by tweaking execution parameters.

9.7 Where to Fix Problems

A critical principle for effective iteration:

Most problems should not be fixed in execution. They should be fixed upstream.

9.7.1 If output is wrong:

Do not immediately: - Change the model or model parameters - Add more prompts or instructions - Adjust temperature or other generation parameters

Instead, ask: - Is the intent clear? Are we solving the right problem? - Is the spec complete? Are we missing important constraints? - Is the instruction precise? Is the execution protocol clear?

Better definition produces better results.

This is a counterintuitive principle. When we see problems, our instinct is often to tweak the execution parameters. But most problems originate in definition, not execution.

9.8 Programming as Spec Iteration

In traditional programming:

  • You write code
  • You debug code—find and fix errors
  • You optimize code—improve performance

The focus is on improving the implementation.

In the LLM paradigm:

  • You define spec
  • You refine spec—improve the definition based on evaluation
  • You iterate spec—systematically improve through evaluation

The focus is on improving the definition.

9.8.1 New Definition

Programming is the process of iteratively refining specifications until the system consistently produces correct results.

This shifts the developer’s activity from:

  • Debugging implementation—finding and fixing code errors

To:

  • Debugging definition—finding and fixing specification gaps

9.9 Stability Through Iteration

A system becomes stable when:

  • Outputs are consistent across multiple runs
  • Behavior matches expectations reliably
  • Evaluation passes consistently

This stability is not achieved through:

  • One-time design—getting it right the first time

But through:

  • Repeated refinement—systematic improvement through iteration

9.9.1 Convergence Condition

A system is considered stable when:

  • Outputs consistently meet success criteria
  • Variations are within acceptable bounds
  • Further iterations produce diminishing returns

Stability is an emergent property that comes from good iteration practices.

9.10 Common Mistakes

9.10.1 1. Skipping Evaluation

  • Leads to false confidence—you think the system works but it doesn’t
  • Systems appear to work but fail in edge cases
  • No basis for improvement

Evaluation is not optional. It is essential.

9.10.2 2. Evaluating Informally

  • No clear criteria—evaluation is subjective and variable
  • Results cannot be compared—no baseline for improvement
  • Feedback is vague—“it doesn’t feel right”

Evaluation must be systematic and explicit.

9.10.3 3. Fixing the Wrong Layer

  • Adjusting execution instead of improving spec
  • Tweaking parameters instead of clarifying definitions
  • Wasting effort on the wrong problems

Most problems originate in definition, not execution.

9.10.4 4. Over-Iterating Without Direction

  • Changing things randomly without evaluation
  • No clear evaluation signal to guide improvements
  • Iteration becomes noise rather than progress

Iteration must be guided by evaluation.

9.11 Evaluation as Feedback Signal

Evaluation provides the signal that drives improvement. Without evaluation, there is no learning, no convergence, no systematic progress.

Without evaluation: - There is no learning—system doesn’t know what’s working - There is no convergence—no way to approach optimal behavior - There is no improvement—changes are random rather than purposeful

With evaluation: - Systems can evolve—learning from feedback - Definitions become sharper—clarifying through iteration - Outputs improve over time—systematic progress toward goals

Evaluation turns iteration into progress.

Without evaluation, iteration is just change. With evaluation, iteration becomes improvement.

9.12 Final Insight

Execution produces outputs. Evaluation determines value. Iteration produces improvement.

Together, they form the dynamic part of the system—the mechanism that enables systems to get better over time.

The quality of a system is not determined by how well it executes, but by how well it evaluates and iterates.

9.13 Closing Statement

In the LLM era, programming is no longer about writing correct code once. It is about refining imperfect definitions until they become reliable.

You do not debug the system by changing what it does. You debug it by clarifying what it should do.

The ability to design effective evaluation and iteration processes is now a core programming skill.

9.14 Key Insight

Evaluation is the process of determining whether output meets intent and spec. It can be rule-based, human, model-based, or hybrid. Iteration guided by evaluation is essential for systematic improvement.

9.15 Action Hint

For every system you build, define explicit evaluation criteria before execution. Ask: “How will I know if this works?” Make evaluation criteria measurable and testable. Don’t skip evaluation—it’s what separates systems that work from systems that appear to work.

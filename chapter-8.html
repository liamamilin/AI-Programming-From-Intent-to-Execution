9  Evaluation & Iteration

9.1 Why Execution Is Not the End

In traditional programming, execution often marks completion:

  • The code runs

  • The system produces output

  • The task is considered done

In the LLM paradigm, this assumption no longer holds.

Execution produces a result, not a guarantee.

Because:

  • Outputs are probabilistic

  • Behavior depends on input variation

  • Systems interpret, rather than strictly follow instructions

Therefore:

Execution is not the end of the process.
It is the beginning of validation.

9.2 What is Evaluation?

Evaluation is the process of determining whether the system’s output satisfies the original intent and specification.

It answers:

  • Is the result correct?

  • Does it meet the constraints?

  • Does it achieve the intended outcome?

Evaluation connects output back to definition.

9.3 Why Evaluation Is Necessary

Without evaluation:

  • There is no notion of correctness

  • There is no direction for improvement

  • There is no system stability

9.3.1 In the LLM Era

Because execution is easy:

  • Systems produce outputs quickly

  • Iterations happen frequently

But without evaluation:

  • Iteration becomes random

  • Improvements are accidental

  • Quality cannot be controlled

Speed without evaluation leads to chaos.

9.4 Types of Evaluation

Evaluation can take multiple forms depending on the system.

9.4.1 1. Rule-Based Evaluation

  • Check against explicit constraints

  • Deterministic

Example:

  • Output must be JSON

  • Response time < 2 seconds

9.4.2 2. Human Evaluation

  • Subjective judgment

  • Context-aware

Example:

  • Is the answer helpful?

  • Does it align with user expectations?

9.4.3 3. Model-Based Evaluation

  • Use another model to assess output

Example:

  • Score relevance

  • Detect hallucinations

9.4.4 4. Hybrid Evaluation

  • Combine rules, humans, and models

Strong systems rely on layered evaluation.

9.5 Evaluation Defines Quality

Quality is not inherent in the output.

It is defined by evaluation.

Without evaluation criteria:

  • “Good” becomes subjective

  • Systems drift over time

  • Results cannot be compared

9.5.1 Example

Intent:

“Summarize a document”

Without evaluation:

  • Any summary is acceptable

With evaluation:

  • Must include key points

  • Must be concise

  • Must avoid redundancy

Evaluation makes quality measurable.

9.6 The Role of Iteration

Evaluation alone is not enough.

It must lead to iteration.

9.6.1 Iteration Loop

Define → Execute → Evaluate → Refine

Refinement typically occurs in:

  • Intent (unclear goal)

  • Spec (missing constraints)

  • Instruction (weak structure)

9.7 Where to Fix Problems

A critical insight:

Most problems should not be fixed in execution.

Instead, they should be fixed upstream.

9.7.1 If output is wrong:

Do not immediately:

  • Change the model

  • Add more prompts

  • Adjust temperature

Instead, ask:

  • Is the intent clear?

  • Is the spec complete?

  • Is the instruction precise?

Better definition produces better results.

9.8 Programming as Spec Iteration

In traditional programming:

  • You write code

  • You debug code

In the LLM paradigm:

  • You define spec

  • You refine spec

9.8.1 New Definition

Programming is the process of iteratively refining specifications until the system consistently produces correct results.

This shifts the developer’s activity from:

  • Debugging implementation

To:

  • Debugging definition

9.9 Stability Through Iteration

A system becomes stable when:

  • Outputs are consistent

  • Behavior matches expectations

  • Evaluation passes reliably

This stability is not achieved through:

  • One-time design

But through:

  • Repeated refinement

9.10 Common Mistakes

9.10.1 1. Skipping Evaluation

  • Leads to false confidence

  • Systems appear to work but fail in edge cases

9.10.2 2. Evaluating Informally

  • No clear criteria

  • Results cannot be compared

9.10.3 3. Fixing the Wrong Layer

  • Adjusting execution instead of improving spec

9.10.4 4. Over-Iterating Without Direction

  • Changing things randomly

  • No clear evaluation signal

9.11 Evaluation as Feedback Signal

Evaluation provides the signal that drives improvement.

Without it:

  • There is no learning

  • There is no convergence

With it:

  • Systems can evolve

  • Definitions become sharper

  • Outputs improve over time

Evaluation turns iteration into progress.

9.12 Final Insight

Execution produces outputs.
Evaluation determines value.
Iteration produces improvement.

Together, they form the dynamic part of the system.

9.13 Closing Statement

In the LLM era, programming is no longer about writing correct code once.

It is about refining definitions until correctness emerges.

You do not debug the system by changing what it does.
You debug it by clarifying what it should do.

15  Spec 结构（Spec Structure）

15.1 为什么结构重要

没有结构的 Spec 只是文本——可能包含有用信息但无法可靠指导系统执行的文字和思想的集合。

没有结构的 Spec 可能包含有用的思想，可能准确地描述系统，但无法可靠地指导执行。

要有效，Spec 必须是可解释的、一致的、可复用的。这需要结构。

结构将规格说明转化为可操作的工件。

15.2 没有僵化的结构（Structure Without Rigidity）

一个常见的错误是假设 Spec 必须遵循固定、僵化的格式。这不是必要的——甚至可能适得其反。

不同的系统可能需要不同的细节层次、不同的表现形式、不同的格式。但所有有效的 Spec 共享一个共同的底层结构。格式可能不同，但核心元素是相同的。

目标不是固定的模板，而是一组一致的元素。

15.3 Spec 的核心组件（The Core Components of a Spec）

任何有效的 Spec 必须定义四个核心组件：

15.3.1 输入（Input）——系统接收什么

输入定义系统接收的信息。好的输入定义包括数据类型、格式要求、有效值范围和边界条件。

15.3.2 输出（Output）——系统产生什么

输出定义系统产生的结果。好的输出定义包括数据结构、格式规范、响应时间要求和结果编码。

15.3.3 约束（Constraints）——系统必须尊重什么

约束定义系统在操作时必须遵守的规则。包括行为约束（系统必须与不能做什么）、性能约束（响应时间、吞吐量）、安全约束（认证、授权）和集成约束（与外部系统的兼容性）。

15.3.4 成功标准（Success Criteria）——什么算正确

成功标准定义如何判断系统是否正确工作。包括正确性标准（输出满足要求的条件）、性能标准（在指定时间内）、鲁棒性标准（在变化条件下的一致性）。

这些元素不是可选的。没有全部四个，Spec 就不完整，无法可靠地指导执行。

15.4 Spec 结构模板（Spec Structure Template）

Spec: [名称]

输入:
  - 数据 1: [类型, 格式, 范围]
  - 数据 2: [类型, 格式, 范围]

输出:
  - 结果 1: [类型, 格式, 结构]
  - 结果 2: [类型, 格式, 结构]

约束:
  - 行为约束: [规则列表]
  - 性能约束: [时间/资源限制]
  - 安全约束: [认证/授权要求]
  - 集成约束: [兼容性要求]

成功标准:
  - 正确性: [条件]
  - 性能: [指标]
  - 鲁棒性: [条件]

结构将规格从思想转化为可执行的工件。

12  Composition

12.1 From Structure to System

After defining structure—main flow, modules, and data flow—we have a blueprint for how a system should be organized. But a blueprint is not a building. A specification is not a system.

Recall from our exploration of Possibility Space that the Option Space represents “what choices exist at each step.” When we navigate the possibility graph, we face countless decision points. Composition is the art of making those choices effectively.

The next step is:

Composition — combining components into a working system

Structure tells us what exists and how parts relate. Composition determines how those parts actually work together to produce behavior. It is how we navigate the Option Space and make the right choices at each intersection.

Without composition:

  • Modules remain isolated—they exist but don’t interact
  • Capabilities remain unused—potential is unrealized
  • Systems never emerge—we have parts but no whole
  • Possibilities remain unrealized—the Option Space is explored but not actualized

12.2 What is Composition?

Composition is the process of:

Combining smaller units of capability into a larger, functional system

These units can take many forms:

  • Modules—independent components with defined interfaces
  • Functions—reusable pieces of logic
  • Agents—autonomous systems that can reason and act
  • Services—external capabilities accessed through APIs
  • Patterns—repeated structures and approaches

Composition answers fundamental questions:

  • Which components are needed to achieve the goal?
  • How do they connect and communicate?
  • In what order do they operate?

12.3 Capability as a Unit

To understand composition, we must first define a key concept: capability.

12.3.1 What is a Capability?

A capability is something a system can do—a unit of functionality that produces a specific result.

Examples: - Generate text—create written content - Retrieve information—find and fetch data - Store data—save information for later use - Classify input—categorize or label data

Each module, agent, or service provides one or more capabilities.

12.3.2 Example

A language model provides several capabilities:

  • Text generation—create written content
  • Summarization—condense longer texts
  • Reasoning—draw inferences and conclusions

A database provides different capabilities:

  • Storage—save and persist data
  • Retrieval—find and fetch stored data
  • Querying—search and filter data

Systems are built by combining capabilities, not by writing everything from scratch.

12.4 From Components to Systems

A system is not a single capability. It is a composition of capabilities—multiple units working together to achieve a goal that no single capability could accomplish alone.

12.4.1 Example: Chat System

Consider building a chat system. No single capability can provide all the required functionality. We need to combine multiple capabilities:

Capabilities involved: - Input handling—receive and validate user messages - Context storage—maintain conversation history - Response generation—create relevant replies - Output rendering—format and display responses

12.4.2 Composition

User Input
→ Input Handler (capability: input handling)
→ Context Manager (capability: context storage)
→ Response Generator (capability: response generation)
→ Output Renderer (capability: output rendering)

Each component contributes a capability. Together, they form a complete system. None of the components alone could provide the full functionality—the system emerges from their composition.

12.5 Types of Composition

There are several common patterns for combining components into systems. Different patterns create different system behaviors.

Remember that in the Possibility Space, we navigate a graph where each node represents a decision point. These composition patterns represent the different ways we can traverse that graph—the different paths we can choose from the Option Space.

12.5.1 1. Sequential Composition

Components operate in sequence, with each component’s output feeding into the next.

Structure:

A → B → C

Used for: - Pipelines—data processing workflows - ETL—extract, transform, load operations - Step-by-step processing

Example:

Raw Text → Cleaner → Processor → Analyzer → Results

12.5.2 2. Parallel Composition

Components operate simultaneously on different aspects of the problem.

Structure:

A
↘
  → Merge → Output
↗
B

Used for: - Multi-source processing—handling multiple inputs - Aggregation—combining results from different sources - Distributed computing—parallel execution

Example:

Input → [Analyzer A, Analyzer B, Analyzer C] → Aggregate → Result

12.5.3 3. Conditional Composition

Different paths are taken based on conditions or decisions.

Structure:

Input → Decision → A or B

Used for: - Routing—directing inputs to appropriate handlers - Decision systems—choosing between alternatives - Branching logic—if-then-else flows

Example:

Input → Classify → [Text Handler OR Image Handler] → Output

12.5.4 4. Iterative Composition

Components repeat until a condition is met or convergence is achieved.

Structure:

Process → Evaluate → [Repeat if needed]

Used for: - Optimization—iteratively improving results - Refinement loops—progressively clarifying outputs - Convergence algorithms—repeating until stable

Example:

Draft → Evaluate → Refine → [Repeat until quality threshold met]

Different compositions create different system behaviors. The choice of composition pattern is a key design decision.

12.6 Composition in the LLM Era

LLMs introduce new forms of composition that go beyond traditional code-based approaches.

12.6.1 Agent-Based Composition

Multiple agents work together, each with a specialized role and responsibility.

  • Multiple agents with different capabilities
  • Each agent has a defined role
  • Coordinated interaction toward a shared goal

Example:

User Query → Planner Agent → Researcher Agent → Writer Agent → Synthesizer Agent → Output

12.6.2 Tool-Augmented Composition

LLMs are combined with external tools and services to extend their capabilities.

  • LLM + external tools
  • Retrieval systems, APIs, databases
  • Enhanced capabilities through integration

Example:

Query → LLM → [Search Tool, Database API, Calculator] → LLM → Output

12.6.3 Human-in-the-Loop Composition

AI and humans work together in an iterative workflow.

  • AI generates initial output
  • Human evaluates or corrects
  • System refines based on feedback

Example:

AI → Human Review → Corrections → AI Refinement → Final Output

Composition is no longer limited to code. It includes models, tools, and humans working together.

12.7 Interfaces Between Components

For composition to work effectively, components must connect properly. This requires interfaces—well-defined contracts that specify how components interact.

12.7.1 What is an Interface?

An interface defines:

  • Input format—what data a component accepts
  • Output format—what data a component produces
  • Expected behavior—how the component should respond

12.7.2 Example

If one module outputs:

{summary: "Text summary...", key_points: ["Point 1", "Point 2"]}

The next module must accept that exact format as input:

input: {summary: string, key_points: Array<string>}

Composition fails when interfaces are unclear or mismatched.

12.8 Loose vs Tight Coupling

A critical design decision in composition is how tightly components are coupled to each other.

12.8.1 Tight Coupling

Components depend heavily on each other’s internal details.

  • Components are interdependent
  • Hard to modify or replace
  • Changes ripple through the system
  • Difficult to test independently

12.8.2 Loose Coupling

Components interact through well-defined interfaces but remain independent.

  • Clear interfaces define interactions
  • Components are independent
  • Easy to replace or extend
  • Can be tested independently

Good composition favors loose coupling. It makes systems more flexible and maintainable.

12.9 Emergence of Systems

A key insight about composition:

Systems are not explicitly written—they emerge from composition.

When components are properly combined:

  • New behavior appears that no single component possesses
  • Capabilities amplify each other
  • Complexity becomes manageable through structure

12.9.1 Example

Combining three simple capabilities:

  • Retrieval—find relevant information
  • Generation—create written content
  • Memory—store and retrieve past interactions

Creates:

  • A knowledge assistant—can answer questions using retrieved knowledge, remember previous conversations, and generate coherent responses

None of the individual capabilities could do this alone. The knowledge assistant emerges from their composition.

12.10 Common Mistakes

12.10.1 1. Overloading Components

  • One module does too much—violates single responsibility
  • Hard to reuse—tangled capabilities are difficult to extract
  • Hard to maintain—changes affect multiple concerns

Solution: Break large components into smaller, focused ones.

12.10.2 2. Ignoring Interfaces

  • Data mismatch between components
  • Integration failures and errors
  • Systems that don’t work together

Solution: Explicitly define and document all interfaces.

12.10.3 3. Overcomplicating Composition

  • Too many components—unnecessary complexity
  • Unnecessary indirection—components that don’t add value
  • Hard to understand and debug

Solution: Use the simplest composition that achieves the goal.

12.10.4 4. Lack of Clear Flow

  • No defined order of operations
  • Unpredictable behavior
  • Components don’t coordinate properly

Solution: Explicitly define the flow and ensure components follow it.

12.11 Composition as a Skill

Composition is a higher-level skill than implementation. It requires:

  • Understanding capabilities—knowing what each component can do
  • Designing connections—creating effective interfaces
  • Managing interactions—ensuring components work together properly

The better your composition skills:

  • The more powerful your systems—complex behaviors emerge from simple parts
  • The less code you need to write—you reuse and compose existing capabilities
  • The faster you can build—you assemble systems rather than building from scratch

12.12 Final Insight

You do not build systems by writing everything from scratch. You build systems by combining the right parts in the right way.

The power of a system lies not in its components, but in how they are composed.

A master composer can create remarkable systems from simple parts, while a novice might struggle even with sophisticated components. The difference is composition.

12.13 Closing Statement

In the LLM era, programming is increasingly about assembling capabilities into coherent systems.

From components to systems, composition is the bridge.

12.14 Key Insight

Composition combines smaller units of capability into larger systems. Key patterns include sequential, parallel, conditional, and iterative composition. Good composition requires clear interfaces and loose coupling.

12.15 Action Hint

Practice composition by building systems from existing capabilities. Before writing new code, ask: “What capabilities do I already have that I can combine?” Map out the composition pattern and define interfaces clearly.

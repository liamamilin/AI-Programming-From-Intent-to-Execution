20  Instruction Design

20.1 From Execution to Design

Instruction is not just a simple translation from spec to execution. It is not merely a mechanical process of converting requirements into prompts.

Instruction is a design problem—a creative and analytical challenge that determines how well a system will perform.

A poorly designed instruction leads to:

  • Inconsistent outputs—results vary unpredictably
  • Unstable behavior—different runs produce different outcomes
  • Difficult debugging—problems are hard to diagnose and fix

A well-designed instruction leads to:

  • Predictable results—consistent, reliable outputs
  • Reusable components—can be applied across different contexts
  • Scalable systems—can grow and evolve without breaking

Instruction is not something you write once and forget. It is something you design carefully, test systematically, and refine continuously.

20.2 What Makes a Good Instruction?

A good instruction is one that enables a system to operate effectively and reliably. It should empower the system to:

  • Produce correct outputs—results that meet requirements and expectations
  • Maintain consistency—similar inputs produce similar outputs
  • Be reusable across contexts—can be applied to different problems or domains

20.2.1 Core Properties

Effective instructions share several fundamental properties:

20.2.1.1 1. Clarity

  • No ambiguity in task—the purpose is crystal clear
  • Clear expectations—what should happen is obvious
  • No room for interpretation—multiple meanings are eliminated

20.2.1.2 2. Structure

  • Defined input and output—exact formats are specified
  • Explicit format—structure is predictable and consistent
  • Organized logically—information flows naturally

20.2.1.3 3. Constraint

  • Boundaries are enforced—what can and cannot happen is clear
  • Degrees of freedom are controlled—system operates within defined limits
  • Rules are explicit—constraints are stated unambiguously

20.2.1.4 4. Executability

  • The system can act without guessing—no implicit assumptions required
  • No missing information—everything needed is provided
  • Ready for automation—can be executed without human intervention

A good instruction removes the need for interpretation. The system knows exactly what to do.

20.3 Instruction as a Design Artifact

Instruction should be treated with the same care and attention as other design artifacts in software development:

  • An API—defining how components interact
  • A function interface—specifying inputs, outputs, and behavior
  • A contract—establishing clear expectations and obligations

It defines:

  • What goes in—the input the system receives
  • What comes out—the output the system produces
  • How the system behaves—the transformation it performs

20.3.1 Example Analogy

Think of instruction like a function signature in programming:

f(input) → output

The instruction specifies:

  • Input schema—what data the function accepts
  • Output schema—what data the function returns
  • Constraints—rules the function must follow

Good instructions are predictable like functions. Given the same input, they always produce the same output.

20.4 The Importance of Structured Output

One of the most critical aspects of instruction design is output structure. This is often overlooked but has profound implications for system reliability and usability.

20.4.1 Why Structure Matters

Without structured output:

  • Outputs are inconsistent—different runs produce different formats
  • Hard to parse—downstream systems cannot reliably process results
  • Difficult to reuse—results cannot be easily integrated into other systems

With structured output:

  • Outputs are predictable—format is consistent and reliable
  • Easy to process—downstream systems can parse results automatically
  • Compatible with systems—results can be integrated seamlessly

20.5 Types of Output Structure

Output structure exists on a spectrum from completely free-form to highly structured:

20.5.1 1. Free Text (Least Structured)

  • Flexible—system has maximum freedom
  • Hard to control—results vary widely
  • Difficult to parse—format is unpredictable

Example: > “Write a summary of this text.”

The summary could be any length, format, or style.

20.5.2 2. Semi-Structured

  • Some constraints apply—guidelines exist but not strict rules
  • More predictable than free text—format is somewhat consistent
  • Easier to parse—structure is recognizable

Example: > “Write a summary as 3-5 bullet points.”

The format is constrained but bullet points can vary in length and content.

20.5.3 3. Fully Structured

  • Strict format rules—exact structure is specified
  • Highly predictable—format is consistent
  • Easy to parse—structure is machine-readable

Example: > “Output JSON: {summary: string, key_points: Array, word_count: number}”

The format is completely specified and machine-readable.

The more structured the output, the more controllable the system. Structure enables reliability.

20.6 Designing Output First

A useful principle for effective instruction design:

Design the output before designing the instruction

20.6.1 Why?

  • Output defines what success looks like—knowing the end state guides the process
  • Output determines system usability—downstream systems depend on output format
  • Output simplifies instruction—knowing the target makes the path clearer

20.6.2 Example

Instead of starting with: > “Summarize this text”

Start with: - Output format: - Summary: 50-100 words - Key points: 3-5 bullet points - Each bullet: under 20 words

Then design the instruction to achieve this output.

Output design drives instruction clarity. Know where you’re going before you start.

20.7 Balancing Structure and Flexibility

Effective instruction design requires finding the right balance between structure and flexibility.

20.7.1 Too Little Structure

  • Outputs vary widely—results are unpredictable
  • Hard to control—system may produce unexpected results
  • Difficult to integrate—downstream systems cannot use results

20.7.2 Too Much Structure

  • Restricts model capabilities—limits what the system can do
  • Reduces natural generation—results feel artificial or constrained
  • Creates brittleness—small changes break the system

20.7.3 Good Balance

  • Structure where necessary—constrain format and key requirements
  • Flexibility where beneficial—allow freedom in content and style
  • Focus on what matters—constrain critical aspects, allow freedom elsewhere

Constrain format, not creativity. Structure the container, not the content.

20.8 Instruction Granularity

Instructions can operate at different levels of detail and specificity. Choosing the right granularity is a key design decision.

20.8.1 High-Level Instruction

  • Broad tasks—large, complex objectives
  • More flexibility—system has freedom in approach
  • Less control—results may vary

Example: > “Analyze this document and provide insights.”

20.8.2 Low-Level Instruction

  • Specific tasks—narrow, well-defined objectives
  • More control—results are predictable
  • Less adaptable—may not handle variations well

Example: > “Extract all dates from the document and format them as YYYY-MM-DD.”

20.8.3 Trade-off

  • High-level → flexible but less predictable
  • Low-level → predictable but less adaptable

Choose granularity based on system needs. Consider how much control you need versus how much flexibility you want.

20.9 Reusability of Instructions

Well-designed instructions can be reused across different contexts, amplifying their value and reducing development effort.

20.9.1 Characteristics of Reusable Instructions

Reusable instructions are:

  • Clear input/output—interfaces are well-defined
  • Minimal assumptions—don’t depend on specific contexts
  • Modular design—can be combined with other instructions

20.9.2 Example

A reusable instruction for text classification:

  • Input: {text: string, categories: Array<string>}
  • Output: {classification: string, confidence: number}
  • Task: Classify text into one of the provided categories

This can be reused for any classification task by changing the categories.

Reusability increases leverage. Design instructions that can be applied across multiple problems.

20.10 Common Mistakes

Several common mistakes can undermine instruction effectiveness:

20.10.1 1. Treating Instructions as Prompts

  • Informal—lacking structure and precision
  • Inconsistent—varying between executions
  • Unreliable—producing unpredictable results

Solution: Design instructions as structured protocols, not conversational prompts.

20.10.2 2. Ignoring Output Design

  • Leads to unusable results—downstream systems cannot process output
  • Makes integration difficult—results cannot be reliably used
  • Reduces system value—good execution produces bad output

Solution: Always design output first. Know exactly what format you need.

20.10.3 3. Missing Constraints

  • Outputs drift—behavior deviates from requirements over time
  • Quality degrades—results become less accurate or relevant
  • System becomes unreliable—predictability decreases

Solution: Explicitly define all constraints. What must the system do? What must it not do?

20.10.4 4. Overloading Instructions

  • Too many tasks in one step—instruction tries to do too much
  • Reduces clarity—main purpose becomes obscured
  • Makes maintenance difficult—hard to understand and modify

Solution: Keep instructions focused. One clear task per instruction. Use composition for complex workflows.

20.11 Instruction and System Stability

Stable systems depend on stable instructions. If instructions are:

  • Clear—unambiguous and precise
  • Structured—organized and predictable
  • Consistent—same format every time

Then:

  • Outputs become predictable—results meet expectations
  • Behavior stabilizes—system performs reliably
  • Quality improves—consistency enables optimization

Stability is a property of design, not execution. Good instructions produce stable systems.

20.12 Final Insight

Instruction design is the point where:

  • Specification becomes executable—abstract definitions become concrete operations
  • Systems become controllable—behavior becomes predictable and reliable

It is a critical design discipline that determines system effectiveness.

The quality of execution depends on the quality of instruction design.

20.13 Closing Statement

In the LLM era, writing instructions is not about phrasing or crafting clever prompts. It is about structure.

A well-designed instruction does not guide the system. It defines how the system behaves.

20.14 Key Insight

Instruction design is a design problem, not a translation task. Key principles include clarity, structure, constraint, and executability. Design output first, balance structure with flexibility, and focus on reusability.

20.15 Action Hint

Always design your output format before writing instructions. Ask: “What exactly do I want the output to look like?” Define the structure precisely. Then design the instruction to achieve that output. Test with multiple inputs to ensure consistency.

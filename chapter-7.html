8  Instruction: Making It Executable

8.1 From Definition to Action

Intent defines what we want—the human goal and aspiration.

Spec defines what it means—the structured, precise specification of behavior.

But neither of these produces results. A perfectly clear intent and a perfectly precise spec, by themselves, do nothing. They are static descriptions of what should exist, not dynamic processes that create results.

To make a system act, to transform definitions into outcomes, we need one more step:

Instruction — the executable form of definition

Instruction is where a defined system becomes operational. It is the bridge between static definition and dynamic execution.

8.2 What is an Instruction?

An Instruction is a machine-executable task definition. It translates a specification into a form that a system can act upon directly and reliably.

An instruction defines:

  • What task to perform—the specific operation the system should execute
  • What inputs to use—the data and parameters the system should work with
  • What outputs to produce—the format and structure of the results
  • What constraints to follow—the rules and boundaries the system must respect during execution

Instruction is not abstract or descriptive. It is operational. It is the protocol that tells a system exactly how to behave.

If Spec defines the system, Instruction runs the system.

8.3 Instruction vs Prompt

A common and persistent misconception is to equate instruction with prompt. While related, they are fundamentally different concepts that serve different purposes.

8.3.1 Prompt

A prompt is:

  • Informal—conversational and flexible in structure
  • Flexible—designed for exploration and experimentation
  • Often ambiguous—leaves room for interpretation
  • Designed for one-off interaction—intended for single-use conversations

Example: > “Summarize this text in a few sentences.”

This prompt works for casual interaction but lacks the precision needed for reliable system execution.

8.3.2 Instruction

An instruction is:

  • Structured—organized according to a defined format
  • Explicit—leaves nothing implicit or assumed
  • Repeatable—produces consistent results across multiple executions
  • Designed for system execution—intended for automated, reliable operation

Example: > - Task: Summarize the input text > - Input: {text: string} > - Output: {summary: string, key_points: Array<string>} > - Constraints: > - Maximum 5 key points > - Each key point under 20 words > - Summary must include all key points > - Preserve original meaning and tone

This instruction is precise, structured, and executable.

A prompt asks. An instruction defines.

The difference is crucial. Prompts are for conversation and exploration. Instructions are for operation and execution.

8.4 Why Prompts Are Not Enough

Prompts work well for certain contexts:

  • Exploration—trying out ideas and seeing what happens
  • Experimentation—testing hypotheses and gathering information
  • Quick interaction—casual conversations with AI systems

But prompts fail when:

  • Consistency is required—the system must produce reliable, repeatable results
  • Systems must be reliable—behavior must be predictable and trustworthy
  • Outputs must be structured—results need to be processed by other systems

Problems with prompts include:

  • Ambiguity—multiple interpretations are possible
  • Lack of constraints—no boundaries on what the system can produce
  • Unstable results—different runs produce different outputs

Prompts are conversational. Instructions are operational.

For building production systems, we need instructions, not prompts.

8.5 Instruction as Execution Protocol

Instruction should be understood as an execution protocol—a formal set of rules that govern how a system should behave.

A protocol defines:

  • The format of communication—how information is exchanged
  • The rules of interaction—what operations are permitted and how they should be performed
  • The expected behavior—what the system should do in different situations

8.5.1 An Instruction Protocol Typically Includes:

  • Task definition—clear statement of what should be accomplished
  • Input schema—precise specification of input format and structure
  • Output schema—precise specification of output format and structure
  • Constraints—rules that must be followed during execution
  • Optional steps or procedures—algorithms or workflows that should be followed

This protocol-based approach makes the system:

  • Predictable—behavior follows defined rules
  • Reusable—the same instruction can be used multiple times
  • Composable—instructions can be combined into larger systems

8.6 Structure of a Good Instruction

A well-designed instruction has several key components that work together to ensure reliable execution.

8.6.1 1. Task

What should be done? This is the core operation the system must perform.

Example: > “Classify the sentiment of the input text”

The task should be specific and unambiguous.

8.6.2 2. Input

What data is provided? This specifies what the system will work with.

Example: > {text: string, language: string}

The input schema should be precise and include all relevant information.

8.6.3 3. Output

What format is expected? This defines the structure of the results.

Example: > {sentiment: "positive" | "negative" | "neutral", confidence: number}

The output schema should be structured and match the needs of downstream systems.

8.6.4 4. Constraints

What rules must be followed? These are the boundaries within which the system must operate.

Example: > - Use only one sentiment label > Do not add explanation or justification > - Confidence must be between 0 and 1

Constraints ensure that the system produces appropriate results.

8.6.5 5. Evaluation (optional but critical)

How do we know it is correct? This defines how to assess the quality of the output.

Example: > - Output must match one of the predefined categories > - Confidence should correlate with human judgment > - Processing time should be under 1 second

Evaluation criteria provide a basis for improving the instruction.

8.7 Determinism vs Flexibility

Instructions operate in a challenging space between two extremes:

  • Deterministic systems—strict rules that always produce the same output
  • Probabilistic systems—LLMs that generate variable outputs based on probability

8.7.1 Too vague

  • Leads to unpredictable outputs
  • Behavior varies widely between runs
  • System cannot be relied upon for consistent results

8.7.2 Too rigid

  • Reduces the strengths and capabilities of the model
  • Prevents the system from handling novel situations
  • Makes the instruction brittle and fragile

8.7.3 A good instruction

  • Constrains structure—specifies the format and organization of outputs
  • Allows flexibility in generation—lets the model use its capabilities within those structural constraints

Define the boundaries, not every step.

The art of instruction design is finding the right balance between structure and flexibility.

8.8 Instruction as Interface

Instruction is the interface between:

  • Human-defined systems—specs, constraints, and requirements
  • Machine-executed behavior—the actual execution performed by LLMs and agents

It plays a role similar to:

  • APIs in traditional systems—defining how to interact with a component
  • Function signatures in programming—specifying inputs, outputs, and behavior

8.8.1 Example Analogy

In traditional programming:

  • Spec = what a function should do (the contract)
  • Instruction = how to call the function (the invocation)

The spec defines the behavior, while the instruction enables execution.

This interface-based perspective makes instructions:

  • Modular—can be developed, tested, and improved independently
  • Reusable—can be used in multiple contexts and systems
  • Composable—can be combined into larger workflows

8.9 Composing Instructions

Complex systems are not built from a single instruction. They are composed of multiple instructions that work together.

8.9.1 Example

A document processing pipeline may include multiple instructions:

  1. Retrieve data
    • Task: Fetch document from storage
    • Input: {document_id: string}
    • Output: {content: string, metadata: object}
  2. Process data
    • Task: Extract key information
    • Input: {content: string, metadata: object}
    • Output: {entities: Array<object>, topics: Array<string>}
  3. Generate output
    • Task: Create summary
    • Input: {entities: Array<object>, topics: Array<string>}
    • Output: {summary: string, highlights: Array<string>}

Each step:

  • Has its own instruction
  • Produces structured output
  • Feeds into the next step

Systems are networks of instructions.

8.10 Common Mistakes

8.10.1 1. Treating Instructions as Prompts

  • Leads to inconsistent and unpredictable behavior
  • Makes systems unreliable for production use

Instruction requires structure and precision, not conversational flexibility.

8.10.2 2. Ignoring Output Structure

  • Results become hard to use
  • Breaks downstream processes
  • Makes integration difficult

Output structure is essential for building reliable systems.

8.10.3 3. Missing Constraints

  • Outputs drift away from intended behavior
  • Quality degrades over time
  • System becomes unpredictable

Constraints define the boundaries within which the system must operate.

8.10.4 4. Overloading Instructions

  • Trying to do too much in one step
  • Reduces clarity and control
  • Makes debugging and improvement difficult

Each instruction should have a single, clear responsibility.

8.11 Instruction and Control

In traditional programming, control is explicit:

  • Step-by-step execution
  • Precise control over every operation
  • Deterministic behavior

In instruction-based systems, control is indirect:

  • Through constraints—defining boundaries
  • Through structure—specifying formats
  • Through evaluation—measuring and correcting behavior

You do not control every step. You control the shape of execution.

This is a fundamental shift in how we think about control in software systems.

8.12 Final Insight

Instruction is where definition becomes action. It is the point where:

  • Systems stop being conceptual descriptions
  • Systems begin to produce actual results

Without instruction, even the best spec remains a static document. With instruction, specs become operational systems.

8.13 Closing Statement

In the LLM era, writing instructions is not about asking better questions or crafting clever prompts. It is about defining executable protocols.

A well-defined instruction does not request behavior. It produces it.

The ability to design good instructions is now a core skill for system builders.

8.14 Key Insight

Instruction translates specification into executable form. Unlike prompts, instructions are structured, explicit, and designed for reliable system execution. Good instructions balance structure with flexibility.

8.15 Action Hint

Practice writing instructions for simple tasks. Use this format: Task, Input Schema, Output Schema, Constraints. Test your instructions by running them multiple times and checking for consistency. If results vary, your instruction needs more structure.

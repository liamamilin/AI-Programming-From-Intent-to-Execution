24  Building an Agent System

24.1 From Task to System

Up to this point, we have focused on:

  • Single tasks—individual operations or queries
  • Individual instructions—specific protocols for execution
  • Isolated executions—one-off actions without persistence

This approach is useful for:

  • Understanding components—learning how each part works
  • Testing capabilities—verifying that functions work correctly
  • Simple applications—handling straightforward use cases

But real-world applications are not single tasks. They are:

  • Multi-step—requiring sequences of operations
  • Stateful—maintaining information across interactions
  • Interactive—responding to user input and feedback

This requires a fundamental shift:

From solving one task → to building a system that continuously solves tasks

24.2 What is an Agent System?

An Agent System is a system that:

Continuously receives tasks, processes them through defined structures, and produces reliable outcomes

It is not:

  • A single prompt—a one-time request to a model
  • A one-time execution—a single operation that completes and ends
  • An isolated process—a standalone function with no connection to others

It is:

  • A structured pipeline—a sequence of processing stages
  • With memory—preserving information across time
  • With tools—accessing external capabilities
  • With coordination—managing multiple components and flows

24.3 The Evolution Path

Building an agent system follows a natural progression, from simple to complex. Understanding this path helps design systems incrementally, adding capabilities as needed.

24.3.1 Stage 1 — Single Task

  • One input—a single query or request
  • One output—a single response or result
  • No memory—each execution is independent

Example: - Summarize a document - Answer a question - Classify text

Use case: - Simple, one-off operations - Testing capabilities - Prototyping functionality

24.3.2 Stage 2 — Structured Task

  • Defined spec—clear specification of requirements
  • Structured instruction—explicit protocol for execution
  • Controlled output—predictable, consistent results

Example: - Summarize with specific constraints (length, format, content) - Answer with reference to sources - Classify with confidence scores

Use case: - More reliable single tasks - Production-ready individual operations - - Components of larger systems

24.3.3 Stage 3 — Multi-Step Task

  • Multiple instructions—sequence of operations
  • Sequential flow—each step builds on previous
  • Intermediate outputs—results passed between steps

Example: - Retrieve documents → Summarize → Format → Output - Analyze → Classify → Report → Store - Extract → Transform → Load (ETL)

Use case: - Complex workflows - Data processing pipelines - - Multi-stage reasoning

24.3.4 Stage 4 — Stateful System

  • Maintains context—preserves information across executions
  • Uses memory—stores and retrieves data over time
  • Adapts behavior—changes based on history

Example: - Conversation system that remembers previous turns - Personalized assistant that learns preferences - - Adaptive system that improves with use

Use case: - Interactive applications - Long-running processes - - Personalized experiences

24.3.5 Stage 5 — Full Agent System

  • Uses tools—accesses external APIs and services
  • Handles multiple tasks—supports various operations
  • Iterates and adapts—improves through feedback

Example: - Virtual assistant with search, calculation, and reasoning capabilities - - Research agent that can query databases, analyze data, and generate reports - - Autonomous agent that can plan, execute, and evaluate tasks

Use case: - Production systems - - Complex applications - - Autonomous operation

Task → Structured Task → Multi-Step → Stateful → Agent System

Each stage adds capabilities, transforming from a simple function to a sophisticated system.

Systems evolve through layering capabilities. Start simple, add complexity as needed.

24.4 Core Components of an Agent System

A complete agent system typically includes several layers, each with specific responsibilities:

24.4.1 1. Input Interface

  • Receives user input—accepts queries, commands, or data
  • Validates format—ensures input is correct and usable
  • Routes to appropriate handler—directs input to right component

Purpose: Ensure clean, valid input enters the system

24.4.2 2. Spec Layer

  • Defines behavior—specifies what the system should do
  • Sets constraints—establishes boundaries and limits
  • Determines success criteria—defines how to evaluate results

Purpose: Provide clear definition and guidance for execution

24.4.3 3. Instruction Layer

  • Converts spec into executable form—translates definition into protocol
  • Manages instruction library—stores and retrieves instructions
  • Handles instruction composition—combines multiple instructions

Purpose: Transform definitions into executable operations

24.4.4 4. Execution Engine

  • Runs instructions—executes the defined operations
  • Coordinates components—manages interaction between system parts
  • Handles tools—invokes external functions and services

Purpose: Actually perform the work defined in instructions

24.4.5 5. Tool Layer

  • External APIs—interfaces to external services
  • Functional capabilities—computational and operational functions
  • Resource access—connections to databases, files, and systems

Purpose: Extend system capabilities beyond internal processing

24.4.6 6. Memory System

  • Stores information—persists data across executions
  • Retrieves context—fetches relevant information for current task
  • Manages state—tracks system status and history

Purpose: Enable continuity and learning across time

24.4.7 7. Evaluation Layer

  • Checks output quality—validates results against requirements
  • Provides feedback—enables system improvement
  • Triggers corrections—handles failures and errors

Purpose: Ensure quality and enable continuous improvement

An agent system is a composition of layers. Each layer has a specific responsibility, working together to produce reliable outcomes.

24.5 The Main Flow of an Agent System

A typical agent system follows a structured flow from input to output:

Input
→ Context Construction
→ Spec / Instruction Selection
→ Execution (with Tools)
→ Output
→ Evaluation
→ Memory Update

Each step is:

  • Defined—has clear purpose and function
  • Structured—follows established protocols
  • Interconnected—relies on and feeds into other steps

24.5.1 Step-by-Step Breakdown

24.5.1.1 1. Input

  • Receive user query or task
  • Validate format and content
  • Route to appropriate handler

24.5.1.2 2. Context Construction

  • Retrieve relevant information from memory
  • Construct context for current task
  • Prepare environment for execution

24.5.1.3 3. Spec/Instruction Selection

  • Identify appropriate spec for task
  • Select or generate corresponding instruction
  • Prepare execution protocol

24.5.1.4 4. Execution

  • Run instruction
  • Use tools as needed
    • Handle intermediate results

24.5.1.5 5. Output

  • Generate response or result
  • Format for user or downstream system
    • Present to user or pass to next stage

24.5.1.6 6. Evaluation

  • Validate output against requirements
  • Check quality and correctness
    • Determine if output is acceptable

24.5.1.7 7. Memory Update

  • Store interaction for future reference
    • Update system state
    • Learn from results

24.6 Designing the System

Building an agent system requires structural thinking and careful design. The process involves:

24.6.1 Step 1: Define the Goal

  • What problem does the system solve?
  • What value does it provide?
  • Who will use it?

Purpose: Establish clear objectives and constraints

24.6.2 Step 2: Define Capabilities

  • What can the system do?
  • What tools does it need?
  • What information must it access?

Purpose: Identify required functionality and resources

24.6.3 Step 3: Define Modules

  • What components are needed?
  • How should they be organized?
  • What are their responsibilities?

Purpose: Design system architecture and component structure

24.6.4 Step 4: Define Flow

  • How do components interact?
  • What is the sequence of operations?
  • How does data flow through the system?

Purpose: Establish clear processes and communication protocols

24.6.5 Step 5: Define Interfaces

  • How do modules communicate?
  • What formats do they use?
  • How are errors handled?

Purpose: Ensure reliable integration between components

Design precedes implementation. A well-designed system is easier to build, test, and maintain.

24.7 From Instructions to Pipelines

A system is not a single instruction. It is a pipeline of instructions—multiple operations working together to achieve complex goals.

24.7.1 Example

User Query
→ Retrieval Instruction (fetch relevant information)
→ Processing Instruction (analyze and extract)
→ Generation Instruction (create response)
→ Formatting Instruction (structure and style output)

Each instruction:

  • Has a clear role—performs a specific function
  • Produces structured output—results in defined format
  • Feeds into next step—output becomes input for subsequent instruction

Pipelines turn instructions into systems. They enable complex, multi-stage processing.

24.8 Adding Memory and Feedback

To make the system adaptive and intelligent, it must incorporate memory and feedback mechanisms:

24.8.1 Memory

  • Store past interactions—preserve conversation history and results
  • Retrieve relevant data—access information that informs current processing
  • Maintain state—track system status and configuration

Purpose: Enable continuity and learning

24.8.2 Feedback

  • Evaluate outputs—assess quality and correctness
  • Identify errors—detect problems and failures
  • Refine behavior—improve instructions and processes

Purpose: Enable continuous improvement

This creates:

  • Continuous improvement—system gets better over time
  • Context-aware behavior—responses are informed by history
  • Adaptive operation—system adjusts based on experience

24.9 Scaling the System

As systems grow in complexity and capability, they must scale effectively:

24.9.1 Add Capabilities

  • More tools—access additional external services
  • More functions—perform a wider range of operations
  • More knowledge—incorporate additional information

24.9.2 Add Agents

  • Specialized roles—agents focused on specific tasks
  • Distributed processing—parallel execution across agents
  • Collaborative operation—agents working together

24.9.3 Improve Spec

  • Better definitions—more precise and complete specifications
  • More constraints—clearer boundaries and requirements
  • Better evaluation—more accurate assessment of results

Scaling is achieved by adding structure, not complexity. Well-architected systems scale gracefully.

24.10 Common Mistakes

Several common mistakes undermine effective agent system design:

24.10.1 1. Starting Too Complex

  • Over-engineering early—building more than needed
  • Unnecessary complexity—making systems harder to understand and maintain
  • Wasted effort—building capabilities that aren’t used

Solution: Start simple. Add complexity as needed. Follow the evolution path from single task to full system.

24.10.2 2. Skipping Spec

  • Undefined behavior—system actions are unclear
  • Unstable systems—inconsistent, unpredictable results
  • Difficult debugging—hard to identify and fix problems

Solution: Always define spec first. Clear specification leads to reliable systems.

24.10.3 3. Poor Interfaces

  • Data mismatch—incompatible formats between components
  • Integration failures—components can’t communicate effectively
  • System brittleness—changes break existing functionality

Solution: Design clear interfaces. Specify formats, protocols, and error handling.

24.10.4 4. Lack of Evaluation

  • No feedback loop—system can’t learn or improve
  • Unknown quality—don’t know if results are correct
  • Silent failures—errors go undetected

Solution: Implement evaluation at every stage. Check quality, measure performance, and track errors.

24.11 System Thinking vs Task Thinking

The shift from task to system represents a fundamental change in mindset:

24.11.1 Task Thinking

  • Solve one problem—focus on individual operations
  • One-time execution—complete and forget
  • Isolated processes—no connection between operations

Limitations: - Can’t handle complexity - No continuity or learning - - Difficult to scale

24.11.2 System Thinking

  • Design repeatable processes—create sustainable workflows
  • Handle multiple inputs over time—maintain continuity
  • Build for scale—architect for growth and evolution

Benefits: - Handles complex problems - Learns and improves - - Scales effectively

Systems produce value repeatedly. Tasks produce value once.

24.12 Final Insight

An agent system is not built in one step. It is constructed by:

  • Layering capabilities—adding functionality incrementally
  • Connecting components—integrating parts into a coherent whole
  • Refining definitions—improving specs and instructions over time

You do not build systems all at once. You evolve them from simple tasks, adding capabilities as needed, refining structure as you learn.

24.13 Closing Statement

In the LLM era, the goal is not to solve one problem. It is to build systems that solve classes of problems.

From single tasks to systems, structure is what enables scale.

24.14 Key Insight

Agent systems evolve from single tasks through structured tasks, multi-step workflows, stateful systems, to full agent systems. Core components include input interface, spec layer, instruction layer, execution engine, tool layer, memory system, and evaluation layer.

24.15 Action Hint

Follow the evolution path—start simple, add complexity as needed. Design components with clear responsibilities. Define interfaces for communication. Implement evaluation at every stage. Add memory and feedback for continuous improvement. Build for scale by adding structure, not complexity.

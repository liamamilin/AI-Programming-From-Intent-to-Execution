4  编程现在是什么？（What is Programming Now?）

4.1 必须重新追问的问题

如果编程的核心过程已经发生了根本性的变化——如果执行被外包了，定义成了瓶颈——那么一个更基本的问题必须被重新审视：

编程是什么？

几十年来，这个问题有一个清晰、隐含的答案。程序员、计算机科学家和软件工程师都持有一个很少需要阐述的共同理解。这个理解是：写代码、实现逻辑、通过精确指令构建软件。

当执行是主要约束时，这个定义是足够的。当挑战是让计算机做你想做的事时，编程自然意味着以机器可执行的形式提供那些指令。

但在 LLM 时代，这个定义变得不完整且有潜在的误导性。如果执行由 AI 系统处理，如果主要挑战是定义而非实现，那么”编程”到底意味着什么？

4.2 传统定义（The Traditional Definition）

传统上，编程可以相当准确地描述为：

将人类意图通过代码翻译成机器可执行的指令。

这个描述包含几个关键活动：设计算法、编写函数、管理控制流、处理数据结构。

在这个模型中，开发者同时充当翻译者和执行者。他们理解人类意图，弄清楚如何用算法实现，然后编写实现它的代码。认知负荷在理解问题和实现解决方案之间平均分配。

这个模型已经很好地服务了我们几十年。它与编程语言的工作方式、软件的开发方式以及程序员的培训方式相一致。但它始终是当时技术约束的产物。

4.3 为什么这个定义失效了（Why This Definition Breaks Down）

有了 LLM 和 AI 系统，传统模型的几个基本假设不再成立：

  • 代码可以自动生成——AI 系统可以从描述中编写代码
  • 逻辑可以从描述中推断——系统可以弄清楚实现细节
  • 执行可以被委托——智能体可以处理多步骤工作流

这在我们对编程的理解与程序员实际所做的之间创造了一个脱节：

  • 开发者不再需要手动将每个意图细节翻译成代码
  • 系统可以自动填充大部分实现
  • 程序员的作用从做转向定义

结果，写代码的行为不再是编程的核心活动。它变成了一个次要的、通常是自动化的活动。主要活动是完全不同的东西。

4.4 编程即定义（Programming as Definition）

我们现在可以更精确地重新定义编程，以反映新的现实：

编程是定义产生期望结果的系统的过程。

这个定义代表了关注点的根本转变：

  • 从实现到定义
  • 从代码到结构
  • 从步骤到结果

4.4.1 从实现到定义

在传统模型中，编程主要是关于实现——如何让计算机做某事。程序员的心智模型聚焦于执行：写什么代码、用什么算法、用什么数据结构。

在新模型中，编程主要是关于定义——应该存在什么。程序员的心智模型聚焦于结果：期望什么行为、必须遵守什么约束、成功是什么样。

实现成为一个派生的关注点。一旦系统被足够清晰地定义，实现可以自动生成。程序员的价值不在于写代码，而在于编写可以从其生成代码的定义。

4.4.2 从代码到结构

传统上，代码是编程的主要输出。你通过编写功能来评估进展，通过行数来衡量质量，通过语法来衡量技能。

在新模型中，结构取代代码成为主要关注点。重要的是概念的组织方式——组件之间的关系、约束的层级、行为如何组合。

代码成为一种结构化的定义格式，而不是最终目标。结构和定义之间的界限变得模糊。

4.4.3 从步骤到结果

传统编程要求你指定一系列步骤——算法、控制流、指令顺序。系统逐步执行，每个步骤都定义了你想要发生的事情。

新范式允许你更直接地指定结果。你描述期望的最终状态，系统找出如何达到它。关注的焦点从路径转移到目的地。

4.5 一个定义的新框架（A New Framework for Definition）

为了在这个新现实中有效操作，我们需要一个新的心智模型。这个模型基于五个核心概念：

意图（Intent） → 规格说明（Spec） → 指令（Instruction） → 执行（Execution） → 评估（Evaluation）

这是最小循环（Minimal Loop）。它是 LLM 时代编程的基本单元。

在本书的下一部分中，我们将详细探索这个循环的每一部分。但现在，关键点是认识到编程已经有了根本性的转变。它不再仅仅是写代码。它是定义系统——使它们能够被可靠地实现的行为。

编程不再是将想法翻译成代码。它是将需求翻译成定义，然后将定义翻译成结果。

4  What is Programming Now?

4.1 The Question Must Be Reasked

If the core process of programming has changed, then a more fundamental question must be reconsidered:

What is programming?

For decades, the implicit answer has been:

  • Writing code

  • Implementing logic

  • Building software through instructions

This definition was sufficient when execution was the primary constraint.

But in the LLM era, this definition becomes incomplete.

4.2 The Traditional Definition

Traditionally, programming can be described as:

Translating human intent into machine-executable instructions through code.

This involves:

  • Designing algorithms

  • Writing functions

  • Managing control flow

  • Handling data structures

The developer acts as the translator and executor.

4.3 Why This Definition Breaks Down

With LLMs and AI systems:

  • Code can be generated automatically

  • Logic can be inferred from descriptions

  • Execution can be delegated

This creates a disconnect:

  • The developer no longer needs to manually translate every detail

  • The system can fill in large parts of the implementation

As a result:

The act of writing code is no longer the core activity.

4.4 Programming as Definition

We can now redefine programming more precisely:

Programming is the process of defining systems that produce desired outcomes.

This definition shifts focus from:

  • Implementation → Definition

  • Code → Structure

  • Steps → Outcomes

4.5 What Does “Defining a System” Mean?

To define a system is to specify:

  • What the system should do

  • Under what conditions it operates

  • What constraints it must respect

  • How success is evaluated

A system is not just code. It is:

  • Behavior

  • Structure

  • Interaction

  • Boundaries

4.6 Code Becomes a Byproduct

In this new model:

  • Code is still present

  • Systems still execute instructions

However:

Code is no longer the primary artifact. It is a byproduct of definition.

The developer’s role is no longer:

  • “Write the code that does X”

But:

  • “Define X clearly enough that a system can produce it”

4.7 A Shift in Abstraction

This change represents a higher level of abstraction.

4.7.1 Before

  • Think in functions

  • Think in control flow

  • Think in implementation steps

4.7.2 Now

  • Think in systems

  • Think in specifications

  • Think in transformations

4.8 Programming as System Design

Under this lens, programming aligns more closely with system design.

It involves:

  • Modeling problems

  • Structuring components

  • Defining interactions

  • Managing constraints

The developer becomes:

  • A designer of systems

  • A definer of behavior

4.9 The Role of Language

Language becomes central.

Not just programming languages, but:

  • Natural language

  • Structured specifications

  • Hybrid representations

Because:

The system now interprets definitions expressed in language.

This elevates language from a communication tool to an execution interface.

4.10 From Control to Specification

Traditional programming emphasizes control:

  • Explicit instructions

  • Deterministic flows

  • Precise operations

Modern programming emphasizes specification:

  • Desired outcomes

  • Constraints

  • Evaluation criteria

Instead of controlling every step, we define the space within which the system operates.

4.11 Implications for Developers

This redefinition changes what it means to be skilled.

4.11.1 Less emphasis on:

  • Syntax mastery

  • Manual implementation

  • Low-level optimization

4.11.2 More emphasis on:

  • Problem definition

  • Structural clarity

  • Constraint design

  • Evaluation thinking

4.12 A New Mental Model

To adapt, one must shift from:

  • “How do I build this step by step?”

To:

  • “What system, if defined correctly, would produce this outcome?”

4.13 Final Definition

We can now state a refined definition of programming:

Programming is the act of defining systems—through structured intent, specification, and constraints—so that machines can produce desired and verifiable outcomes.

4.14 Closing Insight

This shift does not remove the need for technical knowledge.

It reorganizes it.

The power of a programmer is no longer measured by how much they can implement,
but by how precisely they can define what should exist.

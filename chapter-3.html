4  What is Programming Now?

4.1 The Question Must Be Reasked

If the core process of programming has changed so fundamentally—if execution has been outsourced and definition has become the bottleneck—then a more fundamental question must be reconsidered:

What is programming?

For decades, this question had a clear, implicit answer. Programmers, computer scientists, and software engineers all operated with a shared understanding that rarely needed to be articulated. That understanding was:

  • Writing code
  • Implementing logic
  • Building software through precise instructions

This definition was sufficient when execution was the primary constraint. When the challenge was making the computer do what you wanted, programming naturally meant the act of providing those instructions in a form the machine could execute.

But in the LLM era, this definition becomes incomplete and potentially misleading. If execution is handled by AI systems, and if the primary challenge is definition rather than implementation, then what does “programming” actually mean?

4.2 The Traditional Definition

Traditionally, programming can be described with reasonable accuracy as:

Translating human intent into machine-executable instructions through code.

This description encompasses several key activities:

  • Designing algorithms—figuring out the step-by-step procedures to solve problems
  • Writing functions—implementing reusable pieces of logic
  • Managing control flow—determining the order in which operations occur
  • Handling data structures—organizing and manipulating information

In this model, the developer acts as both translator and executor. They understand the human intent, figure out how to achieve it algorithmically, and then write the code that makes it happen. The cognitive load is spread evenly between understanding the problem and implementing the solution.

This model served us well for decades. It aligned with how programming languages worked, how software was developed, and how programmers were trained. But it was always an artifact of the technological constraints of the time.

4.3 Why This Definition Breaks Down

With LLMs and AI systems, several fundamental assumptions of the traditional model no longer hold:

  • Code can be generated automatically—AI systems can write code from descriptions
  • Logic can be inferred from descriptions—systems can figure out implementation details
  • Execution can be delegated—agents can handle multi-step workflows

This creates a disconnect between our understanding of programming and the reality of what programmers actually do:

  • The developer no longer needs to manually translate every detail of intent into code
  • The system can fill in large parts of the implementation automatically
  • The role of the programmer shifts from doing to defining

As a result, the act of writing code is no longer the core activity of programming. It becomes a secondary, often automated, activity. The primary activity is something else entirely.

4.4 Programming as Definition

We can now redefine programming more precisely in a way that captures the new reality:

Programming is the process of defining systems that produce desired outcomes.

This definition represents a fundamental shift in focus:

  • From Implementation → Definition
  • From Code → Structure
  • From Steps → Outcomes

Let’s examine each of these shifts in detail.

4.4.1 From Implementation to Definition

In the traditional model, programming was primarily about implementation—how to make the computer do something. The programmer’s mental model was focused on execution: what code to write, what algorithms to use, what data structures to employ.

In the new model, programming is primarily about definition—what should exist. The programmer’s mental model focuses on outcomes: what behavior is desired, what constraints must be respected, what success looks like.

The implementation becomes a derived concern. Once the system is defined clearly enough, the implementation can be generated automatically. The programmer’s value lies not in writing code, but in writing definitions that code can be generated from.

4.4.2 From Code to Structure

Code is a specific artifact—a sequence of instructions in a programming language. It is concrete, literal, and machine-readable. But it is also low-level and implementation-focused.

Structure is a higher-level abstraction—it describes the organization and behavior of a system without specifying the implementation details. Structure includes:

  • The components of the system and their relationships
  • The flows of information and control
  • The constraints and boundaries that govern behavior
  • The criteria for evaluating success

In the new model, the programmer works primarily with structure. Code becomes an output of the definition process, not the primary artifact.

4.4.3 From Steps to Outcomes

Traditional programming focuses on steps—what the system should do, in what order, under what conditions. This is a procedural mental model: break the problem down into small pieces, implement each piece, and compose them together.

The new model focuses on outcomes—what the system should produce, regardless of how it achieves it. This is a declarative mental model: describe the desired result, and let the system figure out how to achieve it.

This shift reflects the move from imperative programming (telling the computer what to do) to declarative programming (telling the computer what you want). LLMs make declarative programming practical for a much wider range of problems.

4.5 What Does “Defining a System” Mean?

If programming is defining systems, we need to understand what it means to define a system. A system is more than code—it is a coherent entity that produces behavior.

To define a system is to specify:

  • What the system should do—the functions and capabilities it provides
  • Under what conditions it operates—the context and environment in which it runs
  • What constraints it must respect—the boundaries and limitations it must operate within
  • How success is evaluated—the criteria for determining whether the system works correctly

A system is not just code. It is a combination of:

  • Behavior—what the system does in response to inputs
  • Structure—how the system is organized internally
  • Interaction—how the system relates to other systems and users
  • Boundaries—what the system does not do

When a programmer defines a system, they are specifying all of these aspects at a level that enables automated execution.

4.6 Code Becomes a Byproduct

In this new model, it is important to understand the changing role of code:

  • Code is still present—systems still execute instructions, and those instructions are still expressed as code
  • Systems still execute logic—computers still need to do things, and they still need programs to tell them how

However, the relationship between the programmer and code has fundamentally changed:

Code is no longer the primary artifact. It is a byproduct of definition.

The programmer’s role is no longer:

  • “Write the code that does X”

But rather:

  • “Define X clearly enough that a system can produce code that does it”

Code becomes something that happens as a result of good definition, not something the programmer focuses on directly. The focus is on the definition, and the code emerges from that definition.

This is analogous to how a compiler translates high-level code into machine instructions. The programmer doesn’t write machine code directly; they write high-level code, and the compiler handles the translation. In the LLM era, the programmer writes definitions, and the AI handles the translation to code.

4.7 A Shift in Abstraction

This change represents a move to a higher level of abstraction. To understand this shift, consider what programmers think about in each model:

4.7.1 Before

Programmers thought in terms of implementation details:

  • Think in functions—what code to write, how to organize it
  • Think in control flow—what order operations should happen in
  • Think in implementation steps—how to break a problem down into code

The mental model was procedural and concrete. Programmers had to hold the entire implementation in their heads and ensure it was correct.

4.7.2 Now

Programmers think in terms of system structure:

  • Think in systems—what components are needed and how they relate
  • Think in specifications—what the system should do, not how it does it
  • Think in transformations—what inputs become what outputs

The mental model is structural and declarative. Programmers focus on defining the system at a high level, trusting that the implementation can be generated from that definition.

4.8 Programming as System Design

Under this lens, programming aligns more closely with the discipline of system design than with traditional coding. It involves:

  • Modeling problems—understanding the problem domain and the entities within it
  • Structuring components—organizing the system into coherent, well-defined parts
  • Defining interactions—specifying how components communicate and coordinate
  • Managing constraints—ensuring the system operates within acceptable boundaries

The developer becomes:

  • A designer of systems—architecting behavior rather than implementing it
  • A definer of behavior—specifying what should happen rather than how
  • An orchestrator of capabilities—composing components into coherent wholes

This is a different kind of skill than traditional programming. It requires different aptitudes and different training. But it is no less demanding or valuable.

4.9 The Role of Language

In this new model, language becomes central to programming. Not just programming languages, but language in a broader sense:

  • Natural language—for expressing intent and requirements
  • Structured specifications—for precise definition of system behavior
  • Hybrid representations—for combining natural language with formal structure

Why is language so central? Because:

The system now interprets definitions expressed in language.

This elevates language from a communication tool to an execution interface. The programmer’s ability to use language precisely becomes a core technical skill, not just a soft skill.

This is a significant change. In traditional programming, natural language was used for documentation and comments but not for the actual program. Now, natural language (or structured language) becomes the primary medium for defining systems.

4.10 From Control to Specification

Traditional programming emphasizes control:

  • Explicit instructions—telling the system exactly what to do, step by step
  • Deterministic flows—ensuring that the same input always produces the same output
  • Precise operations—controlling every aspect of execution

Modern programming emphasizes specification:

  • Desired outcomes—describing what the result should be, not how to achieve it
  • Constraints—defining the boundaries within which the system must operate
  • Evaluation criteria—specifying how to determine whether the system succeeded

Instead of controlling every step, we define the space within which the system operates. The system then figures out how to navigate that space to achieve the desired outcome.

This is analogous to the difference between giving someone turn-by-turn directions versus giving them a destination and letting them find their own route. Both get you to the same place, but the approach is fundamentally different.

4.11 Implications for Developers

This redefinition of programming has significant implications for what it means to be skilled and how developers should approach their work.

4.11.1 Less emphasis on:

  • Syntax mastery—knowing the precise syntax of programming languages
  • Manual implementation—writing code line by line
  • Low-level optimization—tweaking code for performance

These skills are still valuable, but they are no longer the primary differentiators. Many aspects of implementation can now be automated.

4.11.2 More emphasis on:

  • Problem definition—clarifying what problem should be solved
  • Structural clarity—organizing systems in coherent, understandable ways
  • Constraint design—specifying boundaries and limitations precisely
  • Evaluation thinking—defining how to measure success and detect failure

These skills are becoming the core competencies of programming in the LLM era.

4.12 A New Mental Model

To adapt to this new paradigm, one must shift from a specific mental model:

From: - “How do I build this step by step?”

To: - “What system, if defined correctly, would produce this outcome?”

This seems like a subtle shift, but it represents a fundamental change in approach. The first question leads to implementation thinking. The second question leads to design thinking.

The effective programmer in the LLM era thinks about systems, not code. They think about outcomes, not steps. They think about definitions, not implementations.

4.13 Final Definition

We can now state a refined definition of programming that captures the new reality:

Programming is the act of defining systems—through structured intent, specification, and constraints—so that machines can produce desired and verifiable outcomes.

This definition emphasizes:

  • Definition as the primary activity
  • Structure as the medium of definition
  • Outcomes as the focus of attention
  • Verification as a critical concern

It moves away from implementation as the core activity and toward definition as the core activity.

4.14 Closing Insight

This shift does not remove the need for technical knowledge or programming skill. It reorganizes and reorients them.

The technical knowledge required is different—more focused on system design, specification, and evaluation, less focused on implementation details and optimization. But it is no less demanding or sophisticated.

The power of a programmer is no longer measured by how much they can implement,
but by how precisely they can define what should exist.

4.15 Key Insight

Programming has evolved from writing code to defining systems. The unit of work has shifted from implementation to specification—clarity of definition is now the primary skill.

4.16 Action Hint

Practice system definition by writing specs before writing code. Describe what you want to build in terms of inputs, outputs, constraints, and success criteria—without thinking about implementation. Only when the spec is clear should you proceed to execution.

18  Iterative Spec Refinement

18.1 Specification Is Not Static

A common misconception in software development is that a specification can be written once and remain correct forever. This misconception has led to countless failed projects and wasted effort.

In reality:

  • Initial specs are incomplete—we never know everything upfront
  • Assumptions are often hidden—we take things for granted without realizing it
  • Edge cases are not fully known—we can’t anticipate every situation

Therefore:

A spec is not a final artifact. It is a starting point.

18.2 The Reality of Imperfect Definitions

Even with careful design, deep domain knowledge, and extensive planning, the first version of a spec will likely have issues:

  • Miss constraints—important limitations are not specified
  • Leave ambiguity—terms or concepts are not clearly defined
  • Fail under certain inputs—edge cases emerge during execution

This is not a failure of the team or the process. It is a natural consequence of:

  • Complex problems—we cannot fully understand everything at the start
  • Limited initial understanding—we learn more as we build and test
  • Evolving requirements—what we need changes as we learn

You do not discover the correct spec before execution. You discover it through execution.

18.3 The Refinement Loop

Spec Engineering is inherently iterative. It is not a linear process of “write spec → implement → done.” It is a cyclical process of continuous improvement.

The core loop is:

Spec → Execution → Deviation → Refinement → Spec

18.3.1 Step 1: Spec

  • Define initial structure—create the first version of the specification
  • Set inputs, outputs, constraints—establish the basic framework
  • Make assumptions explicit—state what you’re taking for granted

18.3.2 Step 2: Execution

  • Run the system—use the spec to guide behavior
  • Produce outputs—generate results based on the spec
  • Observe behavior—see what actually happens

18.3.3 Step 3: Deviation

  • Compare output with expectations—did we get what we wanted?
  • Identify mismatches—where does behavior differ from intent?
  • Recognize patterns—are certain types of errors recurring?

18.3.4 Step 4: Refinement

  • Adjust spec—update the specification based on observations
  • Add constraints—specify boundaries that were missing
  • Clarify definitions—make ambiguous terms precise

Each iteration reduces ambiguity and moves the system closer to the desired behavior.

18.4 What is Deviation?

Deviation is the gap between what we expect and what we actually observe:

Expected Behavior ≠ Actual Output

Understanding the types of deviation helps diagnose where the spec needs improvement.

18.4.1 Types of Deviation

18.4.1.1 1. Missing Information

The output lacks required content or elements.

Example: - Expected: Summary includes all key points - Actual: Summary misses some important points - Diagnosis: Success criteria not specific enough

18.4.1.2 2. Incorrect Interpretation

The system misunderstands the intent or requirements.

Example: - Expected: Formal tone in responses - Actual: Casual language used - Diagnosis: Constraint not clearly specified

18.4.1.3 3. Overgeneralization

The output is too broad or includes unnecessary information.

Example: - Expected: Specific answer to a question - Actual: Long explanation with tangential information - Diagnosis: Output scope not constrained

18.4.1.4 4. Inconsistency

Different runs produce different outputs for similar inputs.

Example: - Expected: Consistent formatting across all responses - Actual: Format varies between runs - Diagnosis: Output structure not specified precisely

Deviation is not noise. It is a signal—a valuable indicator of where the spec needs improvement.

18.5 Where to Apply Fixes

A critical principle for effective refinement:

Fix problems at the highest possible level

When you observe deviation, resist the urge to immediately tweak execution parameters. Instead, look upstream to find the root cause.

18.5.1 Incorrect Approach

  • Change the model—switch to a different AI system
  • Add more prompts—inject additional instructions
  • Adjust parameters—modify temperature, top-k, or other settings

These changes address symptoms, not causes. They may temporarily improve results but don’t solve the underlying problem.

18.5.2 Correct Approach

  • Improve spec—clarify the definition
  • Add constraints—specify missing boundaries
  • Define success criteria—make evaluation more precise

These changes address the root cause. They improve the definition, which leads to better execution.

Most issues originate in definition, not execution.

The vast majority of problems with AI systems are not bugs in the implementation—they are gaps or ambiguities in the specification.

18.6 Refinement Strategies

When you identify deviation, use these strategies to refine the spec effectively:

18.6.1 1. Add Constraints

If outputs are too broad or uncontrolled:

  • Introduce limits—specify maximum length, number of items, etc.
  • Define boundaries—what is included and what is excluded
  • Specify format—exact structure of the output

18.6.2 2. Clarify Output Structure

If outputs are inconsistent or unpredictable:

  • Define format—specify exact structure (JSON, array, etc.)
  • Specify required elements—what must always be included
  • Provide examples—show what good output looks like

18.6.3 3. Define Edge Cases

If the system fails under certain inputs:

  • Add explicit rules—how to handle unusual situations
  • Extend spec—cover more scenarios
  • Provide fallback behavior—what to do when standard approach fails

18.6.4 4. Strengthen Success Criteria

If evaluation is unclear or subjective:

  • Make criteria measurable—use quantitative metrics where possible
  • Reduce subjectivity—make evaluation more objective
  • Provide evaluation examples—show what counts as success

18.7 Convergence of the System

With each iteration, the system should move toward convergence—stable, predictable behavior that meets the requirements.

18.7.1 Convergence Condition

A system is considered stable when:

  • Outputs consistently meet success criteria—95%+ of runs pass evaluation
  • Variations are within acceptable bounds—minor differences don’t matter
  • Behavior is predictable—similar inputs produce similar outputs

18.7.2 Signs of Convergence

  • Deviations become less frequent—errors are rare
  • Refinements become smaller—adjustments are minor tweaks
  • Evaluation scores stabilize—quality metrics stop improving significantly

Stability emerges through iteration. It is not achieved instantly but develops over time.

18.8 From Debugging Code to Debugging Spec

The shift to iterative refinement represents a fundamental change in how we think about debugging and improvement.

18.8.1 In Traditional Programming

  • You debug code—find and fix errors in the implementation
  • Trace execution paths—follow the code to find where things go wrong
  • Fix logic errors—correct mistakes in the algorithm or control flow

18.8.2 In the LLM Paradigm

  • You debug definitions—find and fix gaps in the specification
  • Clarify intent—make sure the goal is understood correctly
  • Adjust constraints—specify boundaries more precisely

18.8.3 Old Model

  • Fix logic errors—correct the implementation
  • Trace execution paths—find where code goes wrong
  • Optimize performance—make code run faster

18.8.4 New Model

  • Fix ambiguity—clarify what is meant
  • Clarify intent—make sure goals are understood
  • Adjust constraints—specify boundaries more precisely

The object of debugging has changed from code to definition.

18.9 Iteration as Learning

Each refinement cycle improves more than just the spec—it improves understanding:

  • The spec—becomes clearer and more complete
  • Your understanding of the problem—deepens with each iteration
  • The system’s behavior—becomes more predictable and reliable

This creates a virtuous feedback loop:

  • Better understanding → better spec—as you learn more, you can define better
  • Better spec → better output—clearer definitions lead to better results
  • Better output → better understanding—seeing what works teaches you more

Iteration is not repetition. It is progressive clarification and improvement.

18.10 Common Mistakes

18.10.1 1. Treating First Output as Final

  • Accepting early results—assuming the first version is good enough
  • Skipping refinement—moving on without improving
  • Settling for “good enough”—not pushing for excellence

Solution: Always assume the first version will need refinement. Plan for iteration from the start.

18.10.2 2. Fixing Symptoms Instead of Causes

  • Adjusting outputs instead of spec—tweaking results without addressing root cause
  • Adding patches instead of fixing definitions—applying band-aid solutions
  • Treating symptoms rather than disease—addressing what you see, not what causes it

Solution: Always look upstream. When you see a problem, ask “what in the spec caused this?”

18.10.3 3. Lack of Evaluation

  • No clear signal for improvement—can’t tell if changes help
  • Subjective assessment—relying on gut feeling rather than data
  • No baseline for comparison—don’t know if you’re getting better

Solution: Define explicit success criteria and measure consistently. Use data to guide refinement.

18.10.4 4. Overreacting to Noise

  • Changing spec based on single outputs—treating outliers as patterns
  • Making large changes for small issues—overcorrecting for minor problems
  • Chasing random variation—trying to fix things that aren’t really problems

Solution: Look for patterns across multiple runs. Don’t change the spec based on single observations.

18.11 Refinement as a Core Skill

Iterative refinement is not optional. It is central to Spec Engineering and effective system development.

It requires:

  • Observation—carefully watching what the system actually does
  • Diagnosis—understanding why deviations occur
  • Controlled adjustment—making targeted, thoughtful changes

Developing this skill takes practice, but it is essential for success in the LLM era.

18.12 Final Insight

A good spec is not written once. It is discovered through iteration.

The process of refinement is not a sign of failure—it is the path to success. Each iteration brings you closer to a specification that produces reliable, predictable results.

The path to correctness is not direct. It is iterative.

18.13 Closing Statement

In the LLM era, programming is no longer about writing perfect code once. It is about refining imperfect definitions until they become reliable.

You do not get the spec right the first time. You make it right over time.

18.14 Key Insight

Spec refinement is an iterative process: Spec → Execution → Deviation → Refinement → Spec. Most issues originate in definition, not execution. Fix problems at the highest possible level by improving the spec.

18.15 Action Hint

Plan for iteration from the start. Don’t expect your first spec to be perfect. Set up a systematic refinement process: execute, evaluate, identify deviations, refine spec, repeat. Keep track of changes and their effects to build understanding over time.

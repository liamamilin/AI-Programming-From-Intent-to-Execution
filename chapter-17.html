18  Iterative Spec Refinement

18.1 Specification Is Not Static

A common misconception is that a specification can be written once and remain correct.

In reality:

  • Initial specs are incomplete

  • Assumptions are often hidden

  • Edge cases are not fully known

Therefore:

A spec is not a final artifact. It is a starting point.

18.2 The Reality of Imperfect Definitions

Even with careful design, the first version of a spec will likely:

  • Miss constraints

  • Leave ambiguity

  • Fail under certain inputs

This is not a failure of the system.

It is a natural consequence of:

  • Complex problems

  • Limited initial understanding

You do not discover the correct spec before execution.
You discover it through execution.

18.3 The Refinement Loop

Spec Engineering is inherently iterative.

The core loop is:

Spec → Execution → Deviation → Refinement → Spec

18.3.1 Step 1: Spec

  • Define initial structure

  • Set inputs, outputs, constraints

18.3.2 Step 2: Execution

  • Run the system

  • Produce outputs

18.3.3 Step 3: Deviation

  • Compare output with expectations

  • Identify mismatches

18.3.4 Step 4: Refinement

  • Adjust spec

  • Add constraints

  • Clarify definitions

Each iteration reduces ambiguity.

18.4 What is Deviation?

Deviation is the gap between:

Expected Behavior ≠ Actual Output

18.4.1 Types of Deviation

18.4.1.1 1. Missing Information

  • Output lacks required content

18.4.1.2 2. Incorrect Interpretation

  • System misunderstands intent

18.4.1.3 3. Overgeneralization

  • Output is too broad

18.4.1.4 4. Inconsistency

  • Different outputs for similar inputs

Deviation is not noise.
It is a signal.

18.5 Where to Apply Fixes

A critical principle:

Fix problems at the highest possible level

18.5.1 Incorrect Approach

  • Adjust model parameters

  • Add more prompts

  • Patch outputs

18.5.2 Correct Approach

  • Improve spec

  • Clarify constraints

  • Define success criteria

Most issues originate in definition, not execution.

18.6 Refinement Strategies

18.6.1 1. Add Constraints

If outputs are too broad:

  • Introduce limits

  • Define boundaries

18.6.2 2. Clarify Output Structure

If outputs are inconsistent:

  • Define format

  • Specify required elements

18.6.3 3. Define Edge Cases

If system fails under certain inputs:

  • Add explicit rules

  • Extend spec

18.6.4 4. Strengthen Success Criteria

If evaluation is unclear:

  • Make criteria measurable

  • Reduce subjectivity

18.7 Convergence of the System

With each iteration:

  • Ambiguity decreases

  • Behavior stabilizes

  • Outputs become predictable

18.7.1 Convergence Condition

A system is considered stable when:

  • Outputs consistently meet success criteria

  • Variations are within acceptable bounds

Stability emerges through iteration.

18.8 From Debugging Code to Debugging Spec

In traditional programming:

  • You debug code

In this paradigm:

  • You debug definitions

18.8.1 Old Model

  • Fix logic errors

  • Trace execution paths

18.8.2 New Model

  • Fix ambiguity

  • Clarify intent

  • Adjust constraints

The object of debugging has changed.

18.9 Iteration as Learning

Each refinement cycle improves:

  • The spec

  • Your understanding of the problem

  • The system’s behavior

This creates a feedback loop:

  • Better understanding → better spec → better output

Iteration is not repetition.
It is progressive clarification.

18.10 Common Mistakes

18.10.1 1. Treating First Output as Final

  • Accepting early results

  • Skipping refinement

18.10.2 2. Fixing Symptoms Instead of Causes

  • Adjusting outputs instead of spec

18.10.3 3. Lack of Evaluation

  • No clear signal for improvement

18.10.4 4. Overreacting to Noise

  • Changing spec based on single outputs

18.11 Refinement as a Core Skill

Iterative refinement is not optional.

It is central to Spec Engineering.

It requires:

  • Observation

  • Diagnosis

  • Controlled adjustment

18.12 Final Insight

A good spec is not written once.

It is discovered through iteration.

The path to correctness is not direct.
It is iterative.

18.13 Closing Statement

In the LLM era, programming is no longer about writing perfect code.

It is about refining imperfect definitions until they become reliable.

You do not get the spec right the first time.
You make it right over time.

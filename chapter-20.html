21  Tool Use & APIs

21.1 From Generation to Action

So far, we have focused on systems that:

  • Process inputs

  • Generate outputs

However, many real-world tasks require more than generation.

They require:

  • Access to external data

  • Interaction with systems

  • Execution of operations

This introduces a new capability:

Tool Use

21.2 What is Tool Use?

Tool use is the ability of an agent to:

Invoke external functions or systems to extend its capabilities

21.2.1 Examples of Tools

  • APIs (weather, search, database)

  • Calculators

  • File systems

  • External services

Without tools, an agent is limited to:

  • Its internal knowledge

  • Its generative ability

With tools, an agent can:

  • Retrieve real-time data

  • Perform computations

  • Interact with the environment

Tools extend the system beyond language.

21.3 Why Tool Use is Necessary

LLMs alone have limitations:

  • No real-time knowledge

  • Limited precision in computation

  • No direct interaction with external systems

Tool use addresses these limitations.

21.3.1 Example

Without tools:

  • “What is the current weather?” → guessed or outdated

With tools:

  • Agent calls weather API → accurate answer

Tools bridge the gap between language and reality.

21.4 Agents as Orchestrators

With tool use, the role of the agent changes.

It becomes:

An orchestrator of capabilities

Instead of:

  • Directly solving every problem

The agent:

  • Decides when to use a tool

  • Selects the appropriate tool

  • Integrates the result

21.4.1 Example Flow

User Query → Agent → Tool Selection → API Call → Result → Final Response

The agent coordinates, rather than performs everything itself.

21.5 What is an API?

An API (Application Programming Interface) is a defined way for systems to interact.

It specifies:

  • What functions are available

  • What inputs they require

  • What outputs they return

21.5.1 Example

Weather API:

  • Input: location

  • Output: temperature, conditions

APIs are:

  • Structured

  • Predictable

  • Machine-readable

APIs are the standard interface for tool use.

21.6 Function Calling

Function calling is a mechanism that allows an agent to:

Select and invoke a predefined function with structured arguments

21.6.1 Key Idea

Instead of generating free text, the model generates:

  • A function name

  • A set of parameters

21.6.2 Example

Function:

get_weather(location)

Agent Output:

call get_weather with {location: "Seattle"}

System executes:

  • Calls API

  • Returns result

Agent continues:

  • Integrates result into response

Function calling makes tool use structured and reliable.

21.7 The Tool Use Loop

Tool use introduces a new loop within execution:

Reason → Select Tool → Call Tool → Observe Result → Continue

This is often referred to as:

  • ReAct (Reasoning + Acting)

21.7.1 Steps

21.7.1.1 1. Reason

  • Understand the task

  • Determine if a tool is needed

21.7.1.2 2. Select Tool

  • Choose appropriate function

21.7.1.3 3. Call Tool

  • Provide structured input

21.7.1.4 4. Observe Result

  • Receive output

21.7.1.5 5. Continue

  • Use result to produce final output

Tool use integrates reasoning with action.

21.8 Designing Tools for Agents

Not all tools are equally usable.

Good tools for agents have:

21.8.1 1. Clear Interface

  • Well-defined inputs and outputs

21.8.2 2. Structured Data

  • Machine-readable formats

21.8.3 3. Minimal Ambiguity

  • No hidden assumptions

21.8.4 4. Focused Functionality

  • One clear purpose per tool

Tools should be designed for machines, not humans.

21.9 Integration with Instruction Design

Tool use must be reflected in instructions.

An instruction should specify:

  • Available tools

  • When to use them

  • How to format calls

21.9.1 Example

  • “If real-time data is required, call the appropriate API.”

  • “Use get_weather for weather queries.”

Instruction defines the tool usage policy.

21.10 Common Mistakes

21.10.1 1. Overusing Tools

  • Calling tools unnecessarily

  • Reduces efficiency

21.10.2 2. Underusing Tools

  • Ignoring available capabilities

  • Leads to inaccurate results

21.10.3 3. Poor Tool Design

  • Ambiguous interfaces

  • Hard to integrate

21.10.4 4. Lack of Error Handling

  • Tool failures not handled

  • System breaks

21.11 Tool Use as Capability Expansion

Tool use fundamentally changes what systems can do.

21.11.1 Without Tools

  • Generate text

  • Perform reasoning

21.11.2 With Tools

  • Interact with real systems

  • Execute actions

  • Access external knowledge

Tool use turns language systems into operational systems.

21.12 Final Insight

Tool use extends the Minimal Loop.

Intent → Spec → Instruction → Execution (with Tools) → Evaluation

Execution is no longer:

  • Just generation

It becomes:

  • Coordination of capabilities

21.13 Closing Statement

In the LLM era, the most powerful systems are not those that generate the best text.

They are those that can:

  • Use the right tools

  • At the right time

  • In the right way

Tool use is how systems move from understanding to action.
